DESCRIPCIÓN

Spark puede ejecutarse sobre distintos gestores de recursos. Standalone es simple y fácil para entornos dedicados; YARN es común en clusters Hadoop; Kubernetes ofrece flexibilidad con contenedores.

  • Standalone: fácil de configurar, buen rendimiento en clústeres dedicados.
  • YARN: integración con Hadoop, multi-tenant.
  • Kubernetes: despliegue en contenedores, portabilidad y orquestación.
  • Mesos: menos usado hoy pero válido para entornos mixtos.

    • En esta lección vamos a introducir las características de cada uno de estos entornos con sus recomendaciones de uso.

    SPARK Standalone

    Spark Standalone

    Spark Standalone es el gestor de clúster nativo de Apache Spark. Permite ejecutar aplicaciones Spark distribuidas en varias máquinas sin depender de YARN o Kubernetes.

    ¿Para qué sirve?

    Standalone facilita montar un clúster Spark de forma rápida y con pocos componentes. Es especialmente útil en entornos pequeños/medios, desarrollo, pruebas y docencia.

    ✅ Proyectos universitarios
    ✅ Laboratorios
    ✅ Pruebas internas
    ✅ Clúster simple on-prem

    Componentes principales

    • Master: coordina el clúster y asigna recursos, decid donde se ejecutan las tareas y asignan los ejecutores a las aplicaciones. Solamente hay un Master, aunque se pueden configurar en alta disponiblidad
    • Workers: nodos donde se ejecutan los procesos de trabajo. Cada worker, ejecuta los procesos llamados "ejecutores", utiliza la memoria y el CPU del nodo. Es el responsable de la ejecución de las tareas Spark.
    • Executors: Son procesos en los workers que ejecutan tareas y mantienen datos en memoria. Envian los resultados al driver
    • Driver: Es la aplicación Spark que se envía al cluster, define el trabjo, divide las tareas y recoge los resultados

    Comandos básicos

    
           #Arrancar el master:
        sbin/start-master.sh

        #Arrancar un worker (conectándolo al master):
        sbin/start-worker.sh spark://IP_MASTER:7077

        #Enviar un job al clúster:
        spark-submit --master spark://IP_MASTER:7077 app.py

    ✅ Ventajas

    • Instalación y puesta en marcha rápida
    • No requiere Hadoop
    • Fácil para desarrollo y docencia

    ⚠️ Limitaciones

    • Menos “ecosistema” que Kubernetes/YARN
    • Gestión de recursos más manual
    • No siempre es el mejor para grandes producciones

    YARN

    En un entorno Big Data tradicional basado en Hadoop, Apache Spark suele ejecutarse utilizando YARN (Yet Another Resource Negotiator) como gestor de recursos del clúster.

    Este modelo se denomina Spark on YARN, y permite ejecutar aplicaciones Spark compartiendo infraestructura con otras herramientas del ecosistema Hadoop. A diferencia del modo Standalone, en este caso Spark no gestiona directamente los recursos, sino que delega esta función en YARN.

    ¿Qué es YARN?

    YARN es el sistema de gestión de recursos introducido en Hadoop 2.x y tiene como objetivo:

    • Asignar CPU y memoria entre aplicaciones
    • Gestionar colas de trabajos y usuarios
    • Ejecutar aplicaciones distribuidas
    • Compartir recursos entre múltiples motores de procesamiento

    Gracias a YARN, un mismo clúster puede ejecutar simultáneamente motores como Spark, MapReduce, Hive, Flink, HBase u otras herramientas.

    ¿Para que sirve?

    Spark YARN se suele utilizar en:

    ✅ Clústeres Hadoop existentes
    ✅ Entornos multi-tenant
    ✅ Grandes organizaciones con infraestructura Hadoop

    ResourceManager (YARN)

    Es el coordinador global del clúster y decide dónde se ejecutan las aplicaciones.

    Funciones principales:

    • Gestión global de recursos
    • Asignación de contenedores
    • Planificación de aplicaciones

    Componentes de YARN

    NodeManager (YARN)

    Se ejecuta en cada nodo del clúster y administra recursos locales y la ejecución de contenedores.

    • Administra recursos locales
    • Ejecuta contenedores de aplicaciones
    • Supervisa procesos en ejecución
    Driver de Spark

    Coordina la ejecución del programa Spark: construye el plan de ejecución, distribuye tareas y recoge resultados. Puede ejecutarse dentro o fuera del clúster.

    Executors de Spark

    Son procesos que ejecutan tareas Spark dentro de contenedores YARN. Procesan datos, mantienen datos en memoria y devuelven resultados al driver.

    Modos de ejecución en YARN

    Client Mode

    El driver se ejecuta en la máquina cliente.

    Ventajas:

    • Facilita la depuración
    • Entorno cómodo para desarrollo

    Desventajas:

    • Si cae el cliente, cae el job
    Cluster Mode

    El driver se ejecuta dentro del clúster.

    Ventajas:

    • Más robusto y adecuado para producción
    • Independiente del cliente

    Es el modo habitual en producción.

    Ventajas del uso de Spark con YARN

    • Integración con infraestructuras Hadoop existentes
    • Compartición de recursos entre múltiples sistemas
    • Gestión de usuarios y colas
    • Uso eficiente de grandes clústeres empresariales

    Limitaciones

    • Dependencia del ecosistema Hadoop
    • Configuración más compleja
    • Menor adopción en algunos entornos cloud modernos
    • Kubernetes se está imponiendo como alternativa en muchos escenarios

    Cuándo se utiliza Spark on YARN

    Este modelo es habitual en:

    • Grandes corporaciones con clústeres Hadoop
    • Plataformas Big Data on-premise
    • Entornos bancarios y telecomunicaciones
    • Data Lakes empresariales tradicionales

    Actualmente muchas organizaciones migran hacia:

    • Spark sobre Kubernetes
    • Servicios cloud gestionados

    Componentes principales

    • ResourceManager: coordina los recursos del clúster y asigna contenedores a las aplicaciones.
    • NodeManager: gestiona los recursos en cada nodo y ejecuta los contenedores asignados por el ResourceManager.
    • ApplicationMaster: es un proceso específico para cada aplicación que negocia recursos con el ResourceManager y supervisa la ejecución de las tareas.

    ✅ Ventajas

    • Integración con Hadoop
    • Gestión de recursos centralizada
    • Multi-tenant

    ⚠️ Limitaciones

    • Requiere clúster Hadoop
    • Más complejo de configurar que Standalone
    • No tan flexible como Kubernetes para contenedores

    Kubernetes

    Kubernetes es un sistema de orquestación de contenedores que se ha convertido en una plataforma popular para ejecutar aplicaciones distribuidas, incluyendo Apache Spark. Spark sobre Kubernetes permite desplegar y gestionar aplicaciones Spark utilizando contenedores, lo que ofrece flexibilidad, portabilidad y escalabilidad.

    ¿Para qué sirve?

    Spark sobre Kubernetes se utiliza en:

    ✅ Entornos cloud nativos
    ✅ Organizaciones con infraestructura Kubernetes
    ✅ Proyectos que buscan portabilidad entre entornos

    Mesos

    Apache Mesos es un gestor de clústeres que permite ejecutar aplicaciones distribuidas en un entorno compartido. Aunque fue popular en algunos entornos, su uso ha disminuido en favor de Kubernetes. Spark sobre Mesos permitía ejecutar aplicaciones Spark utilizando los recursos gestionados por Mesos, pero hoy en día es menos común.

    ¿Para qué sirve?

    Spark sobre Mesos se utilizaba principalmente en:

    ✅ Entornos con infraestructura Mesos existente
    ✅ Proyectos que buscaban una alternativa a YARN antes de la popularización de Kubernetes
