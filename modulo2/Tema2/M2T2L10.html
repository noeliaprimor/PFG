INTRODUCCIÓN

En Apache Spark, el modelo de programación ha evolucionado desde la abstracción inicial basada en RDD hacia un enfoque más declarativo y optimizado mediante DataFrames y Datasets. Mientras que los RDD proporcionan control de bajo nivel sobre transformaciones distribuidas, el API estructurado introduce un nivel de abstracción superior con optimización automática y fuerte integración con SQL.

Desde Spark 2.0, el API estructurado se ha consolidado como el estándar para el desarrollo de pipelines ETL y analítica avanzada, debido a su eficiencia, expresividad y capacidad para adaptar la ejecución a distintos entornos de clúster.

¿Qué es un DataFrame?

Un DataFrame es una colección distribuida de datos organizada en columnas, asociada a un schema (esquema) que define nombres y tipos. Conceptualmente, es comparable a una tabla relacional o una vista SQL, y su diseño permite que Spark optimice la ejecución de forma automática.

Ejemplo básico en PySpark:

df = spark.read.csv("datos.csv", header=True, inferSchema=True)
df.select("nombre", "edad").filter("edad > 30")

¿Qué es un Dataset?

Un Dataset extiende el concepto de DataFrame añadiendo tipado fuerte en lenguajes como Scala y Java. En términos prácticos:

  • Un DataFrame es un Dataset no tipado de filas (Dataset[Row]).
  • Un Dataset tipado existe formalmente en Scala/Java.
  • En PySpark, no se utiliza Dataset tipado como entidad separada; normalmente se trabaja con DataFrames.

Ejemplo en Scala:

case class Persona(nombre: String, edad: Int)
val ds = spark.read.json("personas.json").as[Persona]

Principios fundamentales del API estructurado

Programación declarativa

El desarrollador describe qué resultado desea, no cómo ejecutarlo físicamente. Spark decide la estrategia de ejecución: 

df.groupBy("departamento").count()
Optimización automática

Mediante el optimizador Catalyst, Spark puede reordenar operaciones, empujar filtros hacia el origen (cuando el conector lo soporta), eliminar columnas innecesarias y optimizar joins.

Planificación lógica y física

Cada operación se traduce a un conjunto de planes:

  1. Logical Plan
  2. Optimized Logical Plan
  3. Physical Plan
  4. Ejecución distribuida (stages y tasks)

Ventajas frente a RDD

Característica RDD DataFrame/Dataset
Nivel de abstracción Bajo Alto
Optimización automática No Sí
Integración SQL No Sí
Plan lógico/físico No Sí
Rendimiento típico en ETL/Analítica Medio Alto
Casos de uso típicos
  • Procesos ETL (limpieza, normalización, enriquecimiento).
  • Analítica exploratoria y agregaciones masivas.
  • Preparación de datasets para BI.
  • Procesamiento batch y streaming estructurado.

Relación con el motor interno

Aunque el desarrollador trabaja a alto nivel, la ejecución final se materializa como tareas sobre particiones distribuidas: 

Dataset → DataFrame → Logical Plan → Physical Plan → RDD → Tasks

Conclusión

El API de DataFrames y Datasets representa la evolución del modelo de programación de Spark hacia un paradigma declarativo, optimizado y orientado al procesamiento de datos estructurados a gran escala. En entornos modernos de Big Data, constituye el estándar recomendado para desarrollar aplicaciones escalables y eficientes.

EJEMPLOS AVANZADOS DEL API DE DATAFRAMES(PYSPARK) Y DATASETS (SCALA/JAVA)

Ventanas avanzadas (Window Functions)

Top-N por grupo (row_number / dense_rank): Selección de los tres empleados con mayor salario 
from pyspark.sql import functions as F
from pyspark.sql.window import Window

w = Window.partitionBy("departamento").orderBy(F.desc("salario"))

df_top = (df
  .withColumn("rn", F.row_number().over(w))
  .filter(F.col("rn") <= 3)
  .drop("rn")
)
Métricas móviles (rolling) con rangeBetween: Calcula el promedio del campo importe en los últimos 7 días para cada user_id, tomando como referencia cada fila (en este caso, ts debe ser tipo timestamp o convertible a long) 
from pyspark.sql import functions as F
from pyspark.sql.window import Window

w7d = (Window
  .partitionBy("user_id")
  .orderBy(F.col("ts").cast("long"))
  .rangeBetween(-7*24*3600, 0)
)

df_roll = df.withColumn("avg_7d", F.avg("importe").over(w7d))
Sessionization con lag + sum acumulado : agrupa los eventos de cada user_id en sesiones, considerando que empieza una sesión nueva cuando pasan más de 30 minutos entre un evento y el siguiente. Si un usuario genera eventos con marcas de tiempo, ts, este código:
  • Ordena esos eventos por tiempo
  • Calcula cúanto timpo ha pasado desde el evento anterior
  • Marca el inicio de una nueva sesión si el tiempo transcurrido es mayor a 30 minutos o si no hay evento anterior (primer evento del usuario)
  • Asigna un session_id acumulado
from pyspark.sql import functions as F
from pyspark.sql.window import Window

w = Window.partitionBy("user_id").orderBy("ts")

df_sess = (df
  .withColumn("prev_ts", F.lag("ts").over(w))
  .withColumn("gap_min", (F.col("ts").cast("long") - F.col("prev_ts").cast("long"))/60)
  .withColumn("new_sess", F.when(F.col("prev_ts").isNull() | (F.col("gap_min") > 30), 1).otherwise(0))
  .withColumn("session_id", F.sum("new_sess").over(w))
  .drop("prev_ts", "gap_min", "new_sess")
)

Joins avanzados

Anti join / Semi join
  • left_anti: Toma los elemento de la izquierda que NO están en la derecha
  • left_semi: Toma los elementos de la izquierda que SÍ están en la derecha

Recuerda que un inner join normal, devuelve columnas de ambos DataFrames y puede duplicar filas

# Usuarios que NO tienen transacciones (left_anti)
df_no_tx = users.join(tx.select("user_id").distinct(), "user_id", "left_anti")

# Usuarios que SÍ tienen transacciones (left_semi)
df_has_tx = users.join(tx.select("user_id").distinct(), "user_id", "left_semi")
Join con condición no equi-join (rangos) : Join por rango temporal entre las transacciones Tx y promociones. Se asocia a cada transacción la promoción que estaba activa en ese momento. Es decir, ocurre despúes o en el inicio de la promo >= start_ts y antes del final < end_ts 
cond = (
  (tx["ts"] >= promo["start_ts"]) &
  (tx["ts"] <  promo["end_ts"])
)
tx_promo = tx.join(promo, cond, "left")
Join optimizado con broadcast: Es una buena práctica cuando tienes una tabla grande y otra pequeña, se recomienda forzar el broadcast de la tabla pequeña para evitar shuffles costosos. En este ejemplo, se asume que dim_productos es una dimensión pequeña que se puede replicar en cada nodo para evitar shuffle. 
from pyspark.sql import functions as F

tx_enriched = tx.join(F.broadcast(dim_productos), "product_id", "left")

Agregaciones avanzadas: pivot, rollup, cube, grouping sets

Pivot: Permite transformar filas en columnas. En este ejemplo, se calcula la suma de ventas por país y canal, pivotando el campo canal para obtener columnas separadas para cada tipo de canal (web, mobile, store). 
from pyspark.sql import functions as F

df_pivot = (df
  .groupBy("pais")
  .pivot("canal", ["web", "mobile", "store"])
  .agg(F.sum("importe").alias("ventas"))
)
Rollup / Cube: Permiten generar agregaciones a múltiples niveles de granularidad. Rollup genera subtotales jerárquicos, mientras que Cube genera todas las combinaciones posibles. En este ejemplo, se calcula la suma de ventas por país y ciudad (rollup) y por país y canal (cube). El resultado de rollup incluirá filas con total por país ciudad = null y total general pais = null, ciudad = null. El resultado de cube incluirá filas con total por país canal = null, total por canal pais = null y total general pais = null, canal = null. 
from pyspark.sql import functions as F

roll = df.rollup("pais", "ciudad").agg(F.sum("importe").alias("ventas"))
cube = df.cube("pais", "canal").agg(F.sum("importe").alias("ventas"))
Grouping sets vía SQL: Permite definir explícitamente las combinaciones de agrupación que se desean calcular. En este ejemplo, se calculan las ventas por país y ciudad, por país y canal, y por país sin desglosar. 
df.createOrReplaceTempView("t")
spark.sql("""
SELECT pais, ciudad, canal, SUM(importe) ventas
FROM t
GROUP BY GROUPING SETS (
  (pais, ciudad),
  (pais, canal),
  (pais)
)
""")

Manejo avanzado de estructuras complejas (arrays, maps, structs)

explode / posexplode (arrays): Permite convertir elementos de un array en filas individuales. En este ejemplo, se toma una columna tags que es un array de etiquetas y se genera una fila por cada etiqueta, manteniendo el resto de columnas. 
from pyspark.sql import functions as F

df_ex = (df
  .withColumn("tag", F.explode("tags"))
)
transform / filter / size en arrays (expresiones): Permite aplicar transformaciones y filtros directamente sobre arrays sin necesidad de explotar la columna. En este ejemplo, se crean nuevas columnas a partir de la columna tags: una con las etiquetas en mayúscula, otra con solo las etiquetas que empiezan con 'A', y otra con el número total de etiquetas. 
from pyspark.sql import functions as F

df_arr = (df
  .withColumn("tags_upper", F.expr("transform(tags, x -> upper(x))"))
  .withColumn("tags_filtradas", F.expr("filter(tags, x -> x like 'A%')"))
  .withColumn("n_tags", F.expr("size(tags)"))
)
structs y acceso a campos: Permite agrupar varias columnas en una estructura anidada y acceder a sus campos. En este ejemplo, se crea una nueva columna persona que es un struct con los campos nombre y edad, y luego se accede a esos campos para seleccionarlos. 
from pyspark.sql import functions as F

df_struct = df.select(
  "id",
  F.struct("nombre", "edad").alias("persona")
)

df_struct.select("persona.nombre", "persona.edad")
map_keys / map_values: Permite extraer las claves y valores de una columna de tipo map. En este ejemplo, se toma una columna atributos que es un map (por ejemplo, con clave-valor de características) y se generan dos nuevas columnas: una con las claves (keys) del map y otra con los valores. 
from pyspark.sql import functions as F

df_map = (df
  .withColumn("map_keys", F.map_keys("atributos"))
  .withColumn("map_vals", F.map_values("atributos"))
)

Deduplicación avanzada

En muchos casos, es común encontrarse con datos que contienen registros duplicados según una clave (por ejemplo, un ID de usuario o transacción). En lugar de eliminar todos los duplicados, a menudo se desea mantener solo el registro más reciente o relevante según un criterio específico (por ejemplo, la fecha de actualización). Para lograr esto, se pueden utilizar funciones de ventana para asignar un número de fila ordenado por el campo de timestamp y luego filtrar por el número de fila igual a 1, lo que garantiza que solo se conserve el registro más reciente para cada clave.

Mantener el registro más reciente por clave: Si tienes registros duplicados por una clave (por ejemplo, id) y quieres quedarte solo con el registro más reciente según un campo de timestamp (updated_at), puedes usar funciones de ventana para asignar un número de fila ordenado por fecha y luego filtrar por el número de fila igual a 1. 
from pyspark.sql import functions as F
from pyspark.sql.window import Window

w = Window.partitionBy("id").orderBy(F.desc("updated_at"))

df_latest = (df
  .withColumn("rn", F.row_number().over(w))
  .filter(F.col("rn") == 1)
  .drop("rn")
)
CDC simple (detección de cambios entre snapshots): Si tienes dos DataFrames que representan el estado de una tabla en dos momentos diferentes (por ejemplo, df_old y df_new), puedes realizar un join completo (full outer join) por la clave (por ejemplo, id) y luego usar condiciones para clasificar cada registro como INSERT, DELETE, UPDATE o NOCHANGE. 
from pyspark.sql import functions as F

cols = ["id", "campo1", "campo2"]

old = df_old.select(cols)
new = df_new.select(cols)

changes = (new.alias("n")
  .join(old.alias("o"), "id", "full")
  .withColumn("change_type",
      F.when(F.col("o.id").isNull(), F.lit("INSERT"))
       .when(F.col("n.id").isNull(), F.lit("DELETE"))
       .when(
         F.concat_ws("||", *[F.col(f"n.{c}") for c in cols[1:]]) !=
         F.concat_ws("||", *[F.col(f"o.{c}") for c in cols[1:]]),
         F.lit("UPDATE")
       ).otherwise(F.lit("NOCHANGE"))
  )
)

UDF avanzada (y alternativa recomendada)

Entendemos como UDF a una función definida por el usuario que se registra como función de Spark para ser utilizada en expresiones de DataFrame. Las UDF permiten aplicar lógica personalizada que no está disponible en las funciones integradas de Spark, pero tienen la contra que limitan la optimización del plan de ejecución, ya que Spark no puede entender su lógica interna para reordenar o empujar operaciones. Por eso, siempre que sea posible, es recomendable buscar alternativas nativas utilizando las funciones integradas de Spark, que sí permiten optimización y suelen ser más eficientes.

UDF (limita optimización):En este ejemplo, se define una UDF que asigna un score de 1 si el importe es mayor a 10, y 0 en caso contrario. 
from pyspark.sql import functions as F
from pyspark.sql.types import IntegerType

@F.udf(returnType=IntegerType())
def score(x):
  return 1 if x and x > 10 else 0

df2 = df.withColumn("score", score(F.col("importe")))
Alternativa nativa (recomendada): En lugar de usar una UDF, se puede lograr el mismo resultado utilizando la función condicional when de Spark, que sí permite optimización. 
from pyspark.sql import functions as F

df2 = df.withColumn("score", F.when(F.col("importe") > 10, 1).otherwise(0))

Optimización práctica: AQE, skew y particiones de shuffle

En escenarios de joins con datos muy desbalanceados (skew), donde algunas claves tienen un volumen de datos mucho mayor que otras, es común que el rendimiento se degrade debido a tareas que procesan grandes volúmenes de datos asociados a esas claves. Para mitigar este problema, Spark ofrece la opción de activar el Adaptive Query Execution (AQE) y la optimización específica para skew joins. Además, ajustar el número de particiones de shuffle puede ayudar a mejorar la distribución de la carga y reducir los tiempos de ejecución en operaciones de join y agregación. Es importante inspeccionar el plan de ejecución con explain(True) para entender cómo Spark está ejecutando la consulta y verificar si las optimizaciones están teniendo efecto.

Activar Adaptive Query Execution (AQE) : AQE permite a Spark ajustar dinámicamente el plan de ejecución durante la ejecución de la consulta, lo que puede mejorar el rendimiento en casos de joins con datos skew o cuando el tamaño de las tablas no es conocido previamente. 
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
Ajustar particiones de shuffle : En operaciones de join o agregación que requieren shuffle, el número de particiones de shuffle puede afectar significativamente el rendimiento. Un número demasiado bajo puede generar tareas muy grandes y lentas, mientras que un número demasiado alto puede generar overhead por la gestión de muchas tareas pequeñas. Ajustar este parámetro según el tamaño de los datos y la capacidad del clúster puede ayudar a mejorar los tiempos de ejecución. 
spark.conf.set("spark.sql.shuffle.partitions", "200")
Inspeccionar plan de ejecución: Es fundamental revisar el plan de ejecución para entender cómo Spark está procesando la consulta, identificar posibles cuellos de botella y verificar si las optimizaciones están teniendo efecto. El método explain(True) muestra el plan lógico, optimizado y físico, lo que permite analizar aspectos como el tipo de join utilizado, la presencia de shuffles, la estrategia de agregación, entre otros. 
df_join.explain(True)

Dataset tipado (Scala): operaciones por clave y agregación tipada

En Scala, el API de Datasets permite trabajar con tipos de datos definidos por el usuario (case classes) y realizar operaciones de agrupación y agregación manteniendo el tipo de dato específico. Esto facilita la escritura de código más expresivo y seguro, ya que el compilador puede verificar los tipos en tiempo de compilación.

En este ejemplo, se define una case class Tx para representar una transacción, se carga un Dataset tipado a partir de un archivo Parquet, se agrupa por el campo userId y se calcula la suma de los importes utilizando una función de agregación tipada.

Este enfoque es exclusivo de Scala/Java, ya que en PySpark no existe la distinción entre DataFrame y Dataset tipado, y normalmente se trabaja con DataFrames sin tipado fuerte. 

case class Tx(userId: String, amount: Double, ts: java.sql.Timestamp)

val ds = spark.read.parquet("tx.parquet").as[Tx]

val byUser = ds.groupByKey(_.userId)
val agg = byUser.agg(org.apache.spark.sql.expressions.scalalang.typed.sum[Tx](_.amount))

Patrón avanzado de ETL: Bronze → Silver → Gold

El patrón de arquitectura de datos conocido como Bronze → Silver → Gold es una estrategia comúnmente utilizada en pipelines de datos para organizar y procesar la información de manera eficiente y escalable. Este enfoque se basa en la idea de tener diferentes capas o niveles de procesamiento, donde cada capa tiene un propósito específico:

  • Bronze: Es la capa de ingestión o raw, donde se almacenan los datos tal como llegan, sin transformaciones ni limpieza. Aquí se guardan los datos en su formato original, lo que permite tener un historial completo y la posibilidad de volver a procesar los datos si es necesario.
    • bronze = spark.read.json("s3a://bucket/raw/events/")
  • Silver: Es la capa de procesamiento o limpio, donde se aplican transformaciones, limpieza y normalización a los datos. En esta etapa, se eliminan registros duplicados, se corrigen errores, se convierten tipos de datos y se preparan los datos para su análisis. El objetivo es tener un dataset más estructurado y de calidad, listo para ser utilizado en análisis exploratorios o como base para la capa Gold.
    • from pyspark.sql import functions as F

silver = (bronze
  .withColumn("ts", F.to_timestamp("event_time"))
  .dropna(subset=["user_id", "ts"])
)
  • Gold: Es la capa de consumo o agregado, donde se realizan agregaciones, cálculos complejos y se preparan los datos para su consumo final, como por ejemplo para alimentar dashboards, informes o modelos de machine learning. En esta etapa, se pueden generar tablas agregadas, vistas materializadas o datasets optimizados para consultas de BI, con el objetivo de facilitar el acceso y mejorar el rendimiento en las consultas finales.

    • from pyspark.sql import functions as F

gold = (silver
  .groupBy(F.window("ts", "1 day").alias("day"), "country")
  .agg(F.count("*").alias("events"))
)

Este ejercicio practica el uso de funciones de ventana en Spark. A partir del fichero empleados.json, se obtiene el ranking de empleados por salario dentro de cada departamento.

  2. Define una ventana particionada por departamento.
