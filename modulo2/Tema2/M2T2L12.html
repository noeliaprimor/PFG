EJERCICIO PROPUESTO

Ejercicio práctico comparativo: SparkSession en Standalone vs Kubernetes

El objetivo de este ejercicio es analizar el comportamiento de la creación de una SparkSession y la ejecución de una aplicación simple en dos entornos distintos: modo Standalone y modo Kubernetes.

El objetivo es que el estudiante comprenda cómo cambia la asignación de recursos, el despliegue de executors y el ciclo de vida del Driver según el cluster manager utilizado.

Parte A — Ejecución en modo Standalone

1. Crear una SparkSession apuntando al Master Standalone:

from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("Comparativa-Standalone") \
    .master("spark://spark-master:7077") \
    .config("spark.executor.instances", "2") \
    .config("spark.executor.memory", "1g") \
    .getOrCreate()

2. Generar una carga de trabajo:

rdd = spark.sparkContext.parallelize(range(1, 1000000))
resultado = rdd.map(lambda x: x * 2).filter(lambda x: x % 3 == 0).count()
print("Resultado:", resultado)

3. Analizar:

  • ¿Cuántos executors se crean?
  • ¿En qué Workers se ejecutan?
  • ¿Qué ocurre si se reduce la memoria disponible?
  • ¿Qué sucede si el Master no tiene recursos suficientes?

4. Finalizar la sesión:

spark.stop()
Parte B — Ejecución en modo Kubernetes

1. Crear una SparkSession configurada para Kubernetes:

spark = SparkSession.builder \
    .appName("Comparativa-Kubernetes") \
    .master("k8s://https://kubernetes.default.svc") \
    .config("spark.kubernetes.container.image", "apache/spark:latest") \
    .config("spark.executor.instances", "2") \
    .config("spark.executor.memory", "1g") \
    .config("spark.kubernetes.executor.request.cores", "1") \
    .getOrCreate()

2. Ejecutar la misma carga de trabajo:

rdd = spark.sparkContext.parallelize(range(1, 1000000))
resultado = rdd.map(lambda x: x * 2).filter(lambda x: x % 3 == 0).count()
print("Resultado:", resultado)

3. Analizar:

  • ¿Se crean Pods para cada executor?
  • ¿Cómo decide Kubernetes en qué nodo ejecutar cada Pod?
  • ¿Qué ocurre si el clúster no tiene recursos suficientes?
  • ¿Qué diferencia observas en el aislamiento respecto a Standalone?

Preguntas sobre el ejercicio

  1. ¿Por qué SparkSession no cambia entre ambos entornos aunque el comportamiento interno sí lo haga?
  2. ¿Qué ventajas ofrece Kubernetes en entornos productivos?
  3. ¿En qué escenarios sería preferible utilizar Standalone?
  4. ¿Cómo afecta el cluster manager al ciclo de vida del Driver?
  5. ¿Qué implicaciones tiene cada entorno en términos de resiliencia?

Ejercicio avanzado

Configura autoescalado en Kubernetes (si el entorno lo permite) y analiza cómo varía el número de executors ante un aumento progresivo de la carga (por ejemplo, aumentando el tamaño del dataset). Documenta el comportamiento observado y compáralo con el modo Standalone.

ENUNCIADO

EJERCICIO PROPUESTO

El objetivo de este ejercicio es comparar el impacto de formato y particionamiento en el rendimiento de lectura/escritura mediante un flujo sencillo: CSV → transformación → Parquet particionado.

Requisitos del ejercicio
  • Un fichero CSV con al menos 100.000 registros (puede ser un dataset público o generado).
  • Una columna categórica para particionar (por ejemplo: departamento, pais o categoria).
Pasos
  1. Cargar el CSV con cabecera e inferencia de esquema (solo para la primera carga): 
    df = spark.read \
  .option("header", "true") \
  .option("inferSchema", "true") \
  .csv("datos.csv")
  2. Aplicar una transformación (por ejemplo, filtrar registros o crear una nueva columna): 
    from pyspark.sql.functions import col

df2 = df.filter(col("importe") > 0)
  3. Guardar en Parquet (sin particionar) y medir tamaño en disco: 
    df2.write.mode("overwrite").parquet("salida_parquet_no_part")
  4. Guardar en Parquet particionado por una columna categórica: 
    df2.write \
  .mode("overwrite") \
  .partitionBy("categoria") \
  .parquet("salida_parquet_part")
  5. Comparar tamaños entre el CSV original, el Parquet no particionado y el Parquet particionado.
  6. Comparar tiempos de lectura ejecutando:
    • Lectura completa del dataset
    • Lectura filtrando por una sola categoría/partición

    Ejemplo de lectura filtrando por partición:

    df_part = spark.read.parquet("salida_parquet_part")
df_filtrado = df_part.filter(col("categoria") == "A")
df_filtrado.count()
Entregables
  • Tabla con tamaños (MB) de cada formato.
  • Tabla con tiempos aproximados de lectura (completa vs filtrada).
  • Conclusión breve (5-10 líneas) justificando qué formato/estrategia es más eficiente y por qué.

CRITERIOS DE EVALUACIÓN

  • Funcionalidad: la aplicación procesa los datos correctamente.
  • Documentación: explicas decisiones de particionado y persistencia.
  • Rendimiento: justificas configuraciones y mides tiempos.
