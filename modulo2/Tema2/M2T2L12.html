Ejercicio práctico con Apache Spark sobre un dataset de ventas retail

1. Introducción

En este apartado se propone un ejercicio práctico completo para aplicar distintos conceptos avanzados de Apache Spark sobre un dataset real de ventas retail. El objetivo es trabajar con transformaciones, funciones de ventana, joins especializados, pivotes, enriquecimiento de datos y análisis temporal. https://www.kaggle.com/datasets/berkayalan/retail-sales-data?select=sales.csv

El dataset seleccionado es el fichero sales.csv del conjunto de datos publicado en Kaggle bajo el nombre Retail Sales Data.

El propósito del ejercicio es reproducir un escenario realista de analítica de datos sobre transacciones comerciales, en el que se estudian ventas, comportamiento de clientes, sesiones de compra, promociones y métricas agregadas.

2. Objetivos del ejercicio

Los objetivos principales de este caso práctico son los siguientes:

  • Cargar y preparar un dataset real de ventas en Apache Spark.
  • Calcular métricas de negocio a partir de los datos transaccionales.
  • Aplicar funciones de ventana para rankings, medias móviles y sessionización.
  • Utilizar joins especializados como left_semi, left_anti y broadcast join.
  • Realizar un pivot para construir una tabla analítica por país y canal.
  • Simular la asignación de promociones mediante joins por rango temporal.
  • Interpretar los resultados obtenidos desde una perspectiva analítica y técnica.

3. Descripción general del dataset

El fichero sales.csv contiene información relativa a ventas realizadas en un entorno retail. Aunque la estructura exacta puede variar ligeramente según la versión descargada, el ejercicio parte de una estructura lógica formada por campos como los siguientes:

  • customer_id: identificador del cliente.
  • product_id: identificador del producto.
  • category: categoría del producto.
  • price: precio unitario del producto.
  • quantity: cantidad comprada.
  • timestamp o ts: instante temporal de la transacción.
  • country o pais: país en el que se produce la venta.
  • payment_method o canal: método de pago o canal utilizado.

A partir de estos datos se pueden construir análisis tanto descriptivos como temporales, además de escenarios de enriquecimiento y segmentación.

4. Preparación inicial

Antes de desarrollar el ejercicio, se recomienda cargar el dataset en Spark y realizar una normalización mínima de los nombres de columnas y tipos de datos. 

from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder.appName("RetailAnalysis").getOrCreate()

df = spark.read.csv("sales.csv", header=True, inferSchema=True)

df = (df
    .withColumnRenamed("country", "pais")
    .withColumnRenamed("payment_method", "canal")
    .withColumnRenamed("timestamp", "ts")
)

df = df.withColumn("ts", F.to_timestamp("ts"))

df = df.withColumn("importe", F.col("price") * F.col("quantity"))

En esta fase se crea además la columna importe, que representa el valor monetario de cada transacción y que será utilizada en la mayor parte de los análisis posteriores.

5. Enunciado del ejercicio

Se desea construir un pequeño sistema analítico sobre el dataset de ventas retail para responder a distintas necesidades de negocio y, al mismo tiempo, aplicar técnicas avanzadas de Apache Spark. El ejercicio se divide en varios apartados encadenados.

5.1. Parte 1: Análisis de ventas por país y canal

El primer objetivo consiste en construir una tabla analítica que muestre las ventas totales por país y por canal de compra o método de pago.

Para ello, se debe utilizar un pivot sobre la columna canal.

df_pivot = (df
    .groupBy("pais")
    .pivot("canal", ["web", "mobile", "store"])
    .agg(F.sum("importe").alias("ventas"))
)

Este análisis permite observar la distribución de ventas por país y detectar qué canal domina en cada mercado.

5.2. Parte 2: Top 3 productos por categoría

El segundo objetivo es identificar los tres productos más vendidos dentro de cada categoría. Para ello, primero se agregan las ventas por producto y categoría, y posteriormente se aplica una función de ventana con row_number(). 

from pyspark.sql.window import Window

df_prod = (df
    .groupBy("category", "product_id")
    .agg(F.sum("importe").alias("ventas"))
)

w_top = Window.partitionBy("category").orderBy(F.desc("ventas"))

df_top = (df_prod
    .withColumn("rn", F.row_number().over(w_top))
    .filter(F.col("rn") <= 3)
    .drop("rn")
)

Este apartado permite trabajar con rankings por grupo, una de las aplicaciones más frecuentes de las funciones de ventana en Spark.

5.3. Parte 3: Media móvil de gasto en los últimos 7 días

A continuación, se propone calcular, para cada cliente, el promedio del importe de sus compras realizadas durante los últimos siete días respecto de cada transacción. 

w7d = (Window
    .partitionBy("customer_id")
    .orderBy(F.col("ts").cast("long"))
    .rangeBetween(-7 * 24 * 3600, 0)
)

df_roll = df.withColumn("avg_7d", F.avg("importe").over(w7d))

Esta transformación ilustra cómo utilizar ventanas temporales deslizantes para estudiar el comportamiento reciente de los clientes.

5.4. Parte 4: Sessionización de clientes

El siguiente paso es agrupar las compras de cada cliente en sesiones. Se considerará que comienza una nueva sesión cuando transcurren más de 30 minutos entre una compra y la siguiente. 

w_user = Window.partitionBy("customer_id").orderBy("ts")

df_sess = (df
    .withColumn("prev_ts", F.lag("ts").over(w_user))
    .withColumn("gap_min", (F.col("ts").cast("long") - F.col("prev_ts").cast("long")) / 60)
    .withColumn(
        "new_sess",
        F.when(F.col("prev_ts").isNull() | (F.col("gap_min") > 30), 1).otherwise(0)
    )
    .withColumn("session_id", F.sum("new_sess").over(w_user))
    .drop("prev_ts", "gap_min", "new_sess")
)

Con este procedimiento es posible identificar secuencias de compras relacionadas temporalmente y estudiar el número de sesiones, la duración media o el importe acumulado por sesión.

5.5. Parte 5: Clientes con y sin actividad reciente

Se propone clasificar los clientes en dos grupos:

  • Clientes con transacciones recientes.
  • Clientes sin transacciones recientes.

Para ello, se tomarán como recientes las compras efectuadas en los últimos 30 días y se emplearán joins left_semi y left_anti. 

users = df.select("customer_id").distinct()

tx_recent = df.filter(F.col("ts") >= F.current_timestamp() - F.expr("INTERVAL 30 DAYS"))

df_has_tx = users.join(
    tx_recent.select("customer_id").distinct(),
    "customer_id",
    "left_semi"
)

df_no_tx = users.join(
    tx_recent.select("customer_id").distinct(),
    "customer_id",
    "left_anti"
)

Este apartado es útil para procesos de segmentación, campañas comerciales y detección de clientes inactivos.

5.6. Parte 6: Enriquecimiento de datos mediante broadcast join

En muchos escenarios analíticos existe una tabla de hechos grande y una tabla de dimensiones pequeña. Para reproducir este patrón, se puede construir una pequeña dimensión de productos y unirla con las transacciones usando broadcast. 

dim_productos = df.select("product_id", "category").distinct()

tx_enriched = df.join(F.broadcast(dim_productos), "product_id", "left")

El uso de broadcast join permite reducir el coste de la operación cuando la dimensión es lo suficientemente pequeña como para replicarse en memoria en los ejecutores.

5.7. Parte 7: Asociación de promociones por rango temporal

Otro análisis interesante consiste en asociar a cada transacción la promoción que estaba activa en el instante de compra. Para ello, se puede construir un pequeño DataFrame de promociones con intervalos de tiempo. 

promo = spark.createDataFrame([
    ("promo_1", "2023-01-01 00:00:00", "2023-01-10 00:00:00"),
    ("promo_2", "2023-01-10 00:00:00", "2023-01-20 00:00:00"),
    ("promo_3", "2023-01-20 00:00:00", "2023-01-31 00:00:00")
], ["promo_id", "start_ts", "end_ts"]) \
.withColumn("start_ts", F.to_timestamp("start_ts")) \
.withColumn("end_ts", F.to_timestamp("end_ts"))

cond = (
    (df["ts"] >= promo["start_ts"]) &
    (df["ts"] < promo["end_ts"])
)

df_promo = df.join(promo, cond, "left")

Este tipo de join por rango temporal permite modelar campañas, reglas, periodos promocionales o ventanas de validez de atributos.

5.8. Parte 8: Métricas finales de negocio

Una vez completados los apartados anteriores, se propone calcular un conjunto de indicadores finales que resuman la actividad del dataset.

  • Ventas totales por país.
  • Ventas totales por canal.
  • Ticket medio por cliente.
  • Número de sesiones por cliente.
  • Top categorías por facturación.
  • Clientes sin actividad reciente.
  • Ventas realizadas durante promociones.

Ejemplos de agregaciones posibles:

kpi_pais = df.groupBy("pais").agg(F.sum("importe").alias("ventas_totales"))

kpi_canal = df.groupBy("canal").agg(F.sum("importe").alias("ventas_totales"))

kpi_cliente = df.groupBy("customer_id").agg(
    F.avg("importe").alias("ticket_medio"),
    F.sum("importe").alias("gasto_total")
)

kpi_categoria = df.groupBy("category").agg(
    F.sum("importe").alias("ventas_totales")
).orderBy(F.desc("ventas_totales"))

EJERCICIO PROPUESTO

El objetivo de este ejercicio es comparar el impacto de formato y particionamiento en el rendimiento de lectura/escritura mediante un flujo sencillo: CSV → transformación → Parquet particionado.

Requisitos del ejercicio
  • Un fichero CSV con al menos 100.000 registros (puede ser un dataset público o generado).
  • Una columna categórica para particionar (por ejemplo: departamento, pais o categoria).
Pasos
  1. Cargar el CSV con cabecera e inferencia de esquema (solo para la primera carga): 
    df = spark.read \
  .option("header", "true") \
  .option("inferSchema", "true") \
  .csv("datos.csv")
  2. Aplicar una transformación (por ejemplo, filtrar registros o crear una nueva columna): 
    from pyspark.sql.functions import col

df2 = df.filter(col("importe") > 0)
  3. Guardar en Parquet (sin particionar) y medir tamaño en disco: 
    df2.write.mode("overwrite").parquet("salida_parquet_no_part")
  4. Guardar en Parquet particionado por una columna categórica: 
    df2.write \
  .mode("overwrite") \
  .partitionBy("categoria") \
  .parquet("salida_parquet_part")
  5. Comparar tamaños entre el CSV original, el Parquet no particionado y el Parquet particionado.
  6. Comparar tiempos de lectura ejecutando:
    • Lectura completa del dataset
    • Lectura filtrando por una sola categoría/partición

    Ejemplo de lectura filtrando por partición:

    df_part = spark.read.parquet("salida_parquet_part")
df_filtrado = df_part.filter(col("categoria") == "A")
df_filtrado.count()
Entregables
  • Tabla con tamaños (MB) de cada formato.
  • Tabla con tiempos aproximados de lectura (completa vs filtrada).
  • Conclusión breve (5-10 líneas) justificando qué formato/estrategia es más eficiente y por qué.
