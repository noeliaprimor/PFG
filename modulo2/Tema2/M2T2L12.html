ENUNCIADO

EJERCICIO PROPUESTO

El objetivo de este ejercicio es comparar el impacto de formato y particionamiento en el rendimiento de lectura/escritura mediante un flujo sencillo: CSV → transformación → Parquet particionado.

Requisitos del ejercicio
  • Un fichero CSV con al menos 100.000 registros (puede ser un dataset público o generado).
  • Una columna categórica para particionar (por ejemplo: departamento, pais o categoria).
Pasos
  1. Cargar el CSV con cabecera e inferencia de esquema (solo para la primera carga): 
    df = spark.read \
  .option("header", "true") \
  .option("inferSchema", "true") \
  .csv("datos.csv")
  2. Aplicar una transformación (por ejemplo, filtrar registros o crear una nueva columna): 
    from pyspark.sql.functions import col

df2 = df.filter(col("importe") > 0)
  3. Guardar en Parquet (sin particionar) y medir tamaño en disco: 
    df2.write.mode("overwrite").parquet("salida_parquet_no_part")
  4. Guardar en Parquet particionado por una columna categórica: 
    df2.write \
  .mode("overwrite") \
  .partitionBy("categoria") \
  .parquet("salida_parquet_part")
  5. Comparar tamaños entre el CSV original, el Parquet no particionado y el Parquet particionado.
  6. Comparar tiempos de lectura ejecutando:
    • Lectura completa del dataset
    • Lectura filtrando por una sola categoría/partición

    Ejemplo de lectura filtrando por partición:

    df_part = spark.read.parquet("salida_parquet_part")
df_filtrado = df_part.filter(col("categoria") == "A")
df_filtrado.count()
Entregables
  • Tabla con tamaños (MB) de cada formato.
  • Tabla con tiempos aproximados de lectura (completa vs filtrada).
  • Conclusión breve (5-10 líneas) justificando qué formato/estrategia es más eficiente y por qué.

CRITERIOS DE EVALUACIÓN

  • Funcionalidad: la aplicación procesa los datos correctamente.
  • Documentación: explicas decisiones de particionado y persistencia.
  • Rendimiento: justificas configuraciones y mides tiempos.
