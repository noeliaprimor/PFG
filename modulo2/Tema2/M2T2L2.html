Desarrollo y ejecución de una aplicación Spark

  • Introducción

INTRODUCIÓN

El desarrollo y la ejecución de una aplicación en Apache Spark comprende el conjunto de actividades necesarias para diseñar, implementar, desplegar, ejecutar y monitorizar un procesamiento distribuido de datos. A diferencia de una aplicación tradicional que se ejecuta en una sola máquina, una aplicación Spark está pensada para trabajar sobre un clúster y repartir la carga de trabajo entre varios nodos.

En este contexto, Spark proporciona una arquitectura de ejecución distribuida basada en un Driver, uno o varios Executors y un Cluster Manager, así como una API unificada que permite desarrollar aplicaciones batch, analíticas, de streaming y de machine learning. El punto de entrada recomendado en las versiones modernas de Spark es SparkSession, que unifica el acceso a Spark SQL, DataFrames y otras capacidades del ecosistema.

Esta lección describe de forma estructurada las fases principales del desarrollo y ejecución de una aplicación Spark, desde la definición del problema hasta la monitorización del job en funcionamiento.

CONTENIDO

Concepto de aplicación Spark

Una aplicación Spark es un programa distribuido que se ejecuta sobre un conjunto de recursos gestionados por Spark. En términos generales, una aplicación Spark incluye:

  • Driver Program: proceso principal que contiene la lógica de control de la aplicación.
  • SparkSession / SparkContext: punto de entrada a la funcionalidad de Spark y conexión con el clúster.
  • Executors: procesos distribuidos que ejecutan tareas sobre las particiones de datos.
  • Cluster Manager: componente encargado de asignar recursos a la aplicación.

El Driver define las operaciones sobre los datos, construye el plan de ejecución y coordina el envío de tareas a los Executors. A su vez, los Executors ejecutan dichas tareas y devuelven los resultados parciales o finales al Driver, además de almacenar datos intermedios en memoria o disco cuando es necesario.

Fases del desarrollo de una aplicación Spark

Definición del problema y del tipo de procesamiento

La primera fase del desarrollo consiste en identificar el problema que se desea resolver y determinar el tipo de procesamiento más adecuado. En Spark, una aplicación puede estar orientada a:

  • Procesamiento batch sobre grandes volúmenes de datos históricos.
  • Procesamiento estructurado mediante DataFrames y Spark SQL.
  • Procesamiento de flujos de datos en tiempo real con Structured Streaming.
  • Aplicaciones de aprendizaje automático mediante MLlib.

Esta decisión condiciona tanto la API a utilizar como el diseño del pipeline y la estrategia de ejecución posterior. En desarrollos modernos, el enfoque basado en DataFrames y Spark SQL suele ser el más recomendable debido a sus optimizaciones internas.

Diseño del pipeline de datos

Una aplicación Spark suele organizarse como una secuencia de pasos de procesamiento, a menudo denominada pipeline. De forma general, este pipeline incluye:

  1. Lectura de datos desde una o varias fuentes.
  2. Transformación, limpieza o enriquecimiento de los datos.
  3. Agregación, análisis o aplicación de lógica de negocio.
  4. Escritura del resultado en un sistema de salida.

El diseño del pipeline debe tener en cuenta el formato de entrada, la estructura de los datos, el volumen previsto, el paralelismo necesario y el destino final del resultado.

Elección del lenguaje y de la API

Spark soporta varios lenguajes de programación, principalmente Scala, Python, Java y R. La elección del lenguaje depende del entorno del proyecto, de los conocimientos del desarrollador y del tipo de integración deseada.

Desde el punto de vista de la API, existen distintas opciones:

  • RDD: nivel bajo, más control manual, menos optimización automática.
  • DataFrame: modelo tabular, optimizado, recomendado para la mayoría de casos.
  • Dataset: tipado fuerte en Scala/Java, combina expresividad y optimización.
  • Spark SQL: consultas declarativas sobre datos estructurados.
Inicialización de la aplicación

Una vez definido el diseño general, el primer paso técnico consiste en inicializar la sesión Spark. Esta sesión representa el punto de entrada principal a la aplicación y encapsula la conexión con el clúster y el acceso a las APIs de Spark.

1. Ejemplo en Python 
from pyspark.sql import SparkSession

spark = (
    SparkSession.builder
    .appName("MiAplicacionSpark")
    .getOrCreate()
)
2. Ejemplo en Scala 
import org.apache.spark.sql.SparkSession

val spark = SparkSession
  .builder()
  .appName("MiAplicacionSpark")
  .getOrCreate()

A partir de este objeto se puede acceder tanto a DataFrames como al SparkContext subyacente, en caso de que sea necesario trabajar a un nivel más bajo con RDDs.

Implementación de la lógica de negocio

Una vez creada la sesión, la aplicación implementa la lógica de transformación. Esta fase incluye lectura de datos, filtros, joins, agregaciones, conversiones de tipos, validaciones y escritura del resultado.

1. Ejemplo básico en Python 
df = spark.read.csv("/data/entrada.csv", header=True, inferSchema=True)

resultado = (df
    .filter(df.edad >= 18)
    .groupBy("pais")
    .count())

resultado.write.mode("overwrite").parquet("/data/salida")

En este ejemplo se observa un flujo clásico de aplicación Spark: lectura, transformación y escritura. Aunque el código se expresa de forma secuencial, Spark construye internamente un plan lógico y físico que se ejecutará de forma distribuida cuando se invoque una acción o una escritura.

Pruebas en local

Antes de desplegar la aplicación en un clúster, es habitual ejecutarla en modo local. Esto permite validar la lógica, revisar la salida, detectar errores de esquema y comprobar el comportamiento general del pipeline sin necesidad de utilizar aún recursos distribuidos. 

spark = (
    SparkSession.builder
    .appName("PruebaLocal")
    .master("local[*]")
    .getOrCreate()
)

El modo local resulta especialmente útil en etapas tempranas del desarrollo, en prototipos y en pruebas unitarias o funcionales.

Empaquetado de la aplicación

Empaquetado en Scala o Java

En aplicaciones desarrolladas en Scala o Java, el código suele empaquetarse en un archivo JAR. Este artefacto puede incluir únicamente la aplicación o, de forma más habitual, contener también las dependencias necesarias en un ensamblado o uber jar.

El empaquetado correcto es esencial para garantizar que el Driver y los Executors disponen de todas las clases y bibliotecas requeridas durante la ejecución.

Distribución en Python

En PySpark, la aplicación se distribuye normalmente como un script .py, aunque también pueden añadirse dependencias adicionales mediante archivos .zip, .egg o módulos auxiliares enviados con el comando de ejecución.

Esta fase es importante cuando el proyecto no se limita a un único script, sino que incluye librerías propias, funciones reutilizables o estructuras modulares más complejas.

Ejecución de la aplicación

Uso de spark-submit

El mecanismo estándar para ejecutar aplicaciones Spark es spark-submit. Este comando permite lanzar aplicaciones sobre distintos tipos de clúster, indicando parámetros como el master, la clase principal, el modo de despliegue y opciones de configuración.

1. Ejemplo de ejecución en Python 
spark-submit --master local[*] mi_aplicacion.py
2. Ejemplo de ejecución en Scala 
spark-submit \
  --class com.ejemplo.Main \
  --master spark://spark-master:7077 \
  mi-aplicacion.jar

Mediante este comando, Spark prepara el entorno de ejecución, arranca el Driver, distribuye dependencias y solicita recursos al gestor de clúster correspondiente.

Parámetros principales de ejecución

Algunos de los parámetros más habituales al lanzar una aplicación Spark son los siguientes:

  • --master: define el gestor de recursos o entorno de ejecución.
  • --class: clase principal en aplicaciones Scala/Java.
  • --deploy-mode: determina dónde se ejecuta el Driver.
  • --conf: permite añadir parámetros de configuración específicos.
  • --py-files: distribuye dependencias adicionales en aplicaciones Python.
Modos de ejecución

Spark permite distintos modos de ejecución según el entorno disponible:

  • Local: ejecución en una sola máquina, útil para desarrollo y pruebas.
  • Standalone: modo nativo de Spark, adecuado para laboratorios y clústeres dedicados.
  • YARN: integración con ecosistemas Hadoop.
  • Kubernetes: despliegue en entornos contenerizados modernos.

La elección del modo de ejecución influye en la gestión de recursos, en el despliegue del Driver y en la visibilidad operativa de la aplicación.

Ciclo de ejecución interno de una aplicación Spark

Cuando una aplicación Spark se lanza, no ejecuta inmediatamente todas las instrucciones de transformación. En su lugar, Spark construye primero una representación lógica del trabajo y retrasa la ejecución real hasta que se solicita una acción. Este comportamiento se conoce como evaluación perezosa.

Fases principales
  1. Arranque del Driver y creación de SparkSession.
  2. Conexión con el Cluster Manager y asignación de recursos.
  3. Construcción del plan lógico a partir de las transformaciones.
  4. Optimización del plan de ejecución.
  5. División en jobs, stages y tasks.
  6. Envío de tareas a los Executors.
  7. Ejecución distribuida sobre particiones de datos.
  8. Escritura del resultado o devolución al Driver.
  9. Liberación de recursos al finalizar la aplicación.
Jobs, stages y tasks

Durante la ejecución, Spark divide el trabajo en distintas unidades:

  • Job: unidad global asociada normalmente a una acción.
  • Stage: subconjunto del job que puede ejecutarse sin necesidad de un nuevo shuffle.
  • Task: unidad mínima de trabajo ejecutada sobre una partición.

Esta organización es esencial para comprender el paralelismo interno de Spark y para interpretar la información de la interfaz web de monitorización.

Monitorización de la ejecución

Una parte fundamental del ciclo de vida de una aplicación Spark es su monitorización. Spark ofrece una interfaz web desde el Driver en la que pueden visualizarse los jobs, stages, tasks, executors, uso de almacenamiento, tiempos de ejecución y posibles cuellos de botella.

En un entorno de laboratorio, esta interfaz resulta especialmente útil para:

  • Verificar que el job se ha ejecutado correctamente.
  • Analizar el paralelismo real de la aplicación.
  • Detectar stages lentos o con excesivo shuffle.
  • Observar uso de memoria y distribución de tareas por executor.
Aspectos que conviene analizar
  • Número de jobs generados.
  • Duración de cada stage.
  • Cantidad de tasks por partición.
  • Movimientos de shuffle.
  • Uso de memoria y almacenamiento.
  • Errores y reintentos.

Desarrollo de aplicaciones batch y streaming

Aplicaciones batch

En una aplicación batch, el procesamiento se realiza sobre un conjunto finito de datos. El job empieza, procesa el dataset completo y termina al finalizar la escritura o la acción solicitada.

Aplicaciones streaming

En una aplicación de streaming, por el contrario, el pipeline permanece activo de forma continua a la espera de nuevos eventos. Structured Streaming permite definir esta lógica con una API muy similar a la de batch, pero la ejecución se mantiene en funcionamiento hasta que se detiene explícitamente.

Esta diferencia implica cambios tanto en el diseño de la aplicación como en su despliegue, monitorización y estrategia de tolerancia a fallos.

Buenas prácticas en el desarrollo y ejecución

1. Utilizar SparkSession como punto de entrada

En desarrollos modernos, SparkSession debe ser el punto de entrada principal de la aplicación, ya que integra las capacidades de Spark Core, SQL y DataFrames en una única abstracción.

2. Probar primero en local

La validación inicial en modo local permite reducir errores antes del despliegue en clúster y facilita depuración y pruebas funcionales.

3. Empaquetar correctamente las dependencias

Un empaquetado incompleto puede provocar fallos en tiempo de ejecución, especialmente cuando los Executors necesitan acceder a clases o módulos que no están disponibles en el entorno distribuido.

4. Monitorizar el comportamiento real

No basta con que la aplicación funcione; también debe analizarse su rendimiento, el paralelismo alcanzado y el coste de las operaciones más pesadas, como los joins amplios o el shuffle.

5. Cerrar correctamente la sesión

Al finalizar la aplicación, es recomendable cerrar la sesión de Spark para liberar los recursos asociados: 

spark.stop()

Esta práctica es especialmente importante en entornos Docker, clústeres compartidos y ejecuciones repetidas.

Ejemplo simplificado de aplicación completa en PySpark

from pyspark.sql import SparkSession

spark = (
    SparkSession.builder
    .appName("EjemploAplicacionCompleta")
    .master("local[*]")
    .getOrCreate()
)

df = spark.read.csv("/data/entrada.csv", header=True, inferSchema=True)

resultado = (df
    .filter(df.edad >= 18)
    .groupBy("pais")
    .count())

resultado.write.mode("overwrite").parquet("/data/salida")

spark.stop()

Este ejemplo resume el ciclo habitual de una aplicación Spark: inicialización, lectura, transformación, escritura y finalización.

CONCLUSIÓN

El desarrollo y ejecución de una aplicación Spark es un proceso estructurado que abarca desde la definición del problema hasta la monitorización del job en funcionamiento. Comprender las fases de diseño, las transformaciones, las acciones y el ciclo de ejecución es fundamental para crear aplicaciones eficientes y escalables. Además, la elección de estrategias de optimización, el manejo adecuado de recursos y la observabilidad del rendimiento son aspectos clave para el éxito de cualquier proyecto basado en Spark.
