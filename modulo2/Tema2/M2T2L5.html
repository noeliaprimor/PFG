DESCRIPCIÓN

Una de las funcionalidades más importantes de Apache Spark es su capacidad para leer y escribir datos desde múltiples fuentes de almacenamiento de forma distribuida. Spark no solo procesa datos en memoria, sino que está diseñado para integrarse con diferentes sistemas y formatos, como sistemas de archivos locales, sistemas distribuidos (HDFS), almacenamiento en la nube (S3, Azure Blob, GCS), bases de datos relacionales (vía JDBC) y formatos optimizados como Parquet u ORC.

La lectura y escritura eficiente de datos es clave para el rendimiento y la escalabilidad de cualquier aplicación Big Data, ya que determina tanto el coste computacional como el volumen de datos transferido y almacenado.

Spark utiliza dos componentes principales para interactuar con datos en forma de DataFrames:

  • DataFrameReader: para cargar datos desde una fuente externa.
  • DataFrameWriter: para guardar resultados hacia un destino de almacenamiento.

Ambos se utilizan mediante las propiedades: 

spark.read   # Para cargar
df.write     # Para guardar

El flujo general habitual en escenarios de procesamiento es:

  1. Leer datos desde una fuente externa (ingestión).
  2. Aplicar transformaciones (ETL/ELT).
  3. Escribir los resultados en un almacenamiento destino.

CARGAR DATOS EN SPARK

Lectura de archivos CSV

El formato CSV es común en intercambios de datos, aunque no es el más eficiente para analítica a gran escala. Un ejemplo de lectura: 

df = spark.read \
    .option("header", "true") \
    .option("inferSchema", "true") \
    .csv("datos.csv")

Opciones frecuentes al leer CSV:

Opción Descripción
header Indica si el fichero contiene cabecera con nombres de columnas.
inferSchema Detecta automáticamente tipos de datos (útil en pruebas, más costoso en producción).
delimiter Permite definir un separador distinto de coma (por ejemplo, punto y coma).
quote, escape Controlan caracteres de comillas y escape para casos con texto complejo.

Lectura de JSON

JSON es un formato semiestructurado habitual en APIs y logs. Spark puede inferir el esquema si los registros son consistentes: 

df = spark.read.json("datos.json")

En JSON es especialmente importante mantener consistencia de tipos y estructura para evitar columnas con tipos ambiguos o estructuras anidadas inconsistentes.

Lectura de Parquet

Parquet es un formato columnar optimizado para analítica y es el estándar recomendado en la mayoría de casos Big Data: 

df = spark.read.parquet("datos.parquet")

Ventajas habituales de Parquet:

  • Compresión eficiente (reducción de tamaño en disco).
  • Lectura selectiva de columnas (mejora de rendimiento en consultas).
  • Mejor integración con motores analíticos y catálogos de datos.

Lectura desde bases de datos relacionales (JDBC)

Spark permite leer desde bases de datos mediante JDBC. Ejemplo con PostgreSQL: 

df = spark.read \
    .format("jdbc") \
    .option("url", "jdbc:postgresql://localhost:5432/mi_db") \
    .option("dbtable", "empleados") \
    .option("user", "usuario") \
    .option("password", "password") \
    .load()

Para escalar la lectura y evitar cuellos de botella (lectura serial), es posible paralelizar configurando:

  • partitionColumn
  • lowerBound
  • upperBound
  • numPartitions

Esto permite dividir la tabla en rangos y leerlos en paralelo (siempre que exista una columna adecuada para particionar).

Lectura desde almacenamiento en la nube (ejemplo S3)

Spark puede leer desde almacenamiento en la nube usando conectores adecuados. Ejemplo con S3: 

df = spark.read.parquet("s3a://mi-bucket/datos/")

En entornos cloud, suele requerirse configurar credenciales y endpoints (IAM roles, variables de entorno o secrets).

GUARDAR DATOS EN SPARK

Escritura en CSV

Al escribir en CSV, Spark genera típicamente un directorio con múltiples ficheros part-xxxxx (uno por partición): 

df.write \
    .mode("overwrite") \
    .option("header", "true") \
    .csv("salida_csv")

Escritura en Parquet

Parquet es el formato recomendado para persistir resultados en escenarios analíticos: 

df.write \
    .mode("overwrite") \
    .parquet("salida_parquet")

Escritura en JSON

df.write.json("salida_json")

Escritura en base de datos (JDBC)

Para escribir en una base de datos relacional mediante JDBC: 

df.write \
    .format("jdbc") \
    .option("url", "jdbc:postgresql://localhost:5432/mi_db") \
    .option("dbtable", "empleados_procesados") \
    .option("user", "usuario") \
    .option("password", "password") \
    .mode("append") \
    .save()

Modos de escritura

Spark define diferentes modos de escritura para controlar el comportamiento cuando el destino ya existe:

Modo Descripción
overwrite Sobrescribe los datos existentes (puede borrar el destino).
append Añade nuevos datos al destino.
ignore No realiza ninguna acción si el destino existe.
error / errorifexists Lanza un error si el destino ya existe.

Diferencias entre formatos: CSV vs JSON vs Parquet

La elección del formato impacta en tamaño, rendimiento y coste de almacenamiento/lectura:

Característica CSV JSON Parquet
Estructura Tabular simple Semiestructurado Columnar
Tamaño típico Alto Alto Bajo (con compresión)
Rendimiento analítico Medio Medio Alto
Compresión No nativa No nativa Sí (nativa y eficiente)
Optimizado para Big Data No Parcial Sí

En producción, Parquet suele ser el estándar en procesos de analítica y ETL por su eficiencia y compatibilidad.

Particionamiento al guardar

Spark permite escribir datos particionados en disco, creando una estructura de carpetas por valores de una o varias columnas: 

df.write \
    .partitionBy("departamento") \
    .parquet("salida_particionada")

La salida generada típicamente se organiza como:

salida_particionada/
  departamento=IT/
  departamento=HR/
  ...

Ventajas del particionamiento:

  • Mejora el rendimiento en lecturas filtradas (pruning de particiones).
  • Reduce lectura innecesaria al acceder solo a particiones relevantes.
  • Facilita organización lógica de datasets grandes.

Carga y guardado con RDD

Aunque actualmente el uso de DataFrames es lo más común, Spark también permite cargar y guardar datos con RDD: 

rdd = spark.sparkContext.textFile("datos.txt")
rdd.saveAsTextFile("salida_rdd")

Este enfoque ofrece control de bajo nivel, pero carece de optimizaciones del motor SQL (por ejemplo, Catalyst), por lo que suele reservarse para casos muy específicos.

BUENAS PRÁCTICAS EN EL DESARROLLO DE APLICACIONES SPARK

  • Priorizar formatos columnar (Parquet/ORC) frente a formatos de texto (CSV/JSON) cuando sea posible.
  • Usar inferSchema solo en pruebas; en producción conviene definir esquemas explícitos.
  • Controlar particiones con repartition o coalesce antes de escribir para evitar demasiados ficheros pequeños.
  • Evitar collect() con grandes volúmenes, ya que puede saturar la memoria del Driver.
  • Ajustar spark.sql.shuffle.partitions para operaciones con shuffle según el tamaño real de los datos.
  • En Docker, validar rutas y permisos de volúmenes montados para lectura/escritura.

Conclusión

La carga y guardado de datos en Spark es un componente esencial del procesamiento distribuido. El uso de DataFrameReader y DataFrameWriter permite integrar Spark con múltiples sistemas y formatos, mientras que la elección adecuada del formato (especialmente Parquet) y estrategias como el particionamiento impactan directamente en rendimiento, escalabilidad y costes de almacenamiento.

Dominar estas técnicas es fundamental para desarrollar pipelines Big Data eficientes, reproducibles y alineados con buenas prácticas tanto en entornos académicos como en producción.

Parquet es preferido por su compresión y esquema. El particionado mejora consultas filtradas pero puede generar demasiados ficheros si los buckets son pequeños.

EJERCICIO PROPUESTO

El objetivo de este ejercicio es comparar el impacto de formato y particionamiento en el rendimiento de lectura/escritura mediante un flujo sencillo: CSV → transformación → Parquet particionado.

Requisitos del ejercicio
  • Un fichero CSV con al menos 100.000 registros (puede ser un dataset público o generado).
  • Una columna categórica para particionar (por ejemplo: departamento, pais o categoria).
Pasos
  1. Cargar el CSV con cabecera e inferencia de esquema (solo para la primera carga): 
    df = spark.read \
  .option("header", "true") \
  .option("inferSchema", "true") \
  .csv("datos.csv")
  2. Aplicar una transformación (por ejemplo, filtrar registros o crear una nueva columna): 
    from pyspark.sql.functions import col

df2 = df.filter(col("importe") > 0)
  3. Guardar en Parquet (sin particionar) y medir tamaño en disco: 
    df2.write.mode("overwrite").parquet("salida_parquet_no_part")
  4. Guardar en Parquet particionado por una columna categórica: 
    df2.write \
  .mode("overwrite") \
  .partitionBy("categoria") \
  .parquet("salida_parquet_part")
  5. Comparar tamaños entre el CSV original, el Parquet no particionado y el Parquet particionado.
  6. Comparar tiempos de lectura ejecutando:
    • Lectura completa del dataset
    • Lectura filtrando por una sola categoría/partición

    Ejemplo de lectura filtrando por partición:

    df_part = spark.read.parquet("salida_parquet_part")
df_filtrado = df_part.filter(col("categoria") == "A")
df_filtrado.count()
Entregables
  • Tabla con tamaños (MB) de cada formato.
  • Tabla con tiempos aproximados de lectura (completa vs filtrada).
  • Conclusión breve (5-10 líneas) justificando qué formato/estrategia es más eficiente y por qué.
