DESCRIPCIÓN

En Apache Spark, las colecciones de datos distribuidas constituyen el núcleo del modelo de programación. Spark ofrece tres abstracciones principales para trabajar con datos: RDD (Resilient Distributed Dataset), DataFrame y Dataset. Cada una representa una evolución en términos de nivel de abstracción, optimización y (en ciertos lenguajes) seguridad de tipos, manteniendo como base común la ejecución distribuida sobre particiones.

Comprender en profundidad estas estructuras es esencial para diseñar soluciones Big Data eficientes tanto en entornos académicos como en producción.

RDD (resilient Distributed Dataset)

Un RDD (Resilient Distributed Dataset) es la abstracción fundamental de Spark Core. Representa una colección distribuida e inmutable de elementos que puede procesarse en paralelo.

Características clave:

  • Distribuido: los datos se reparten en múltiples nodos mediante particiones.
  • Inmutable: cada transformación crea un nuevo RDD (no se modifica el original).
  • Evaluación perezosa (lazy evaluation): las transformaciones no se ejecutan hasta que se invoca una acción.
  • Tolerancia a fallos: basada en el linaje (lineage) de transformaciones.

Arquitectura interna de un RDD

Internamente, un RDD puede entenderse como una estructura que describe “cómo obtener” el dataset en lugar de almacenar necesariamente los datos de forma materializada. Sus componentes conceptuales son:

  • Particiones: fragmentos distribuidos de datos.
  • Dependencias: relaciones entre RDDs (dependencias estrechas o amplias).
  • Función de cómputo: transformación aplicada a cada partición.
  • Lineage graph: historial de transformaciones para recomputación en caso de fallo.

La tolerancia a fallos se apoya en el linaje: si una partición se pierde por fallo de un executor o nodo, Spark la recalcula aplicando las transformaciones registradas en el DAG.

Operaciones sobre RDD

Para mas detalle sobre estas operaciones se recomienda revisar la lección 4 del tema 1 del presente módulo aqui.

Transformaciones (evaluación perezosa)
  • map()
  • flatMap()
  • filter()
  • groupByKey()
  • reduceByKey()
Acciones (disparan la ejecución)
  • collect()
  • count()
  • reduce()
  • saveAsTextFile()

Ejemplo básico:

rdd = spark.sparkContext.parallelize([1, 2, 3, 4])
resultado = rdd.map(lambda x: x * 2).filter(lambda x: x > 4).collect()
print(resultado)

Ventajas y limitaciones

Ventajas
  • Control detallado del procesamiento y transformaciones de bajo nivel.
  • Adecuado para algoritmos personalizados y lógica compleja.
  • Útil con datos no estructurados o cuando no existe un esquema claro.
Limitaciones
  • No dispone de optimización automática a nivel de consulta (sin Catalyst).
  • No maneja un esquema de columnas explícito por defecto.
  • Generalmente menor rendimiento en cargas analíticas comunes frente a DataFrames/Datasets.
  • Mayor cantidad de código para operaciones equivalentes.

DATAFRAME

Un DataFrame es una colección distribuida de datos organizada en columnas y con un esquema (schema). Conceptualmente, es similar a una tabla de base de datos o a un DataFrame de Pandas, pero ejecutado de forma distribuida.

Arquitectura interna y optimización

El procesamiento con DataFrames se apoya en Spark SQL y en el optimizador Catalyst. De forma simplificada, una operación sobre DataFrames se transforma en:

  1. Logical plan: representación lógica de la consulta o transformaciones.
  2. Optimización: Catalyst aplica reglas (por ejemplo, empujar filtros, eliminar columnas no usadas, reordenar joins).
  3. Physical plan: plan físico con estrategias de ejecución.
  4. Ejecución: el plan se traduce a operaciones sobre RDD y se ejecuta en el clúster.

Además, Spark utiliza optimizaciones de ejecución y memoria (por ejemplo, mejoras en serialización y representación interna) que suelen traducirse en mayor rendimiento.

Ejemplo de uso

df = spark.read.csv("datos.csv", header=True, inferSchema=True)
df_filtrado = df.filter(df.edad > 30)
df_filtrado.groupBy("departamento").count().show()

Ventajas

  • Optimización automática (Catalyst) y mejores planes de ejecución.
  • API declarativa y expresiva, cercana a SQL.
  • Integración nativa con fuentes de datos y formatos optimizados (Parquet/ORC).
  • Mejor rendimiento en la mayoría de cargas ETL y analíticas.

Limitaciones

  • Menor control de bajo nivel frente a RDD.
  • En PySpark, tipado dinámico (errores de tipo se detectan más tarde).
  • Algunos algoritmos altamente personalizados pueden ser más directos de implementar con RDD.

DATASET

Un Dataset combina ventajas de RDD (tipado fuerte en lenguajes tipados) con las optimizaciones de DataFrame (Catalyst). Está disponible principalmente en Scala y Java.

En términos prácticos: un DataFrame puede considerarse un Dataset “no tipado” (Dataset[Row]). En PySpark, no existe el Dataset tipado como entidad separada; normalmente se trabaja con DataFrames.

Características técnicas

  • Tipado fuerte (Scala/Java): ayuda a detectar errores en compilación.
  • Encoders: mecanismos para serialización/deserialización eficiente de objetos tipados.
  • Optimización Catalyst: conserva el rendimiento y optimizaciones del motor SQL.

Ejemplo en Scala:

case class Persona(nombre: String, edad: Int)

val ds = spark.read.json("personas.json").as[Persona]

Ventajas

  • Mayor seguridad de tipos en proyectos Scala/Java.
  • Detección temprana de errores y refactors más seguros.
  • Rendimiento elevado por optimizaciones de Catalyst y ejecución eficiente.

Limitaciones

  • No disponible como tipado fuerte en Python (PySpark).
  • Mayor complejidad conceptual (encoders y tipado) para ciertos equipos o entornos.

COMPARATIVA

  • RDD: control completo pero sin optimizaciones automáticas.
  • DataFrame: esquema, optimizaciones y API declarativa.
  • Dataset: tipado (Scala/Java) y optimizaciones.

Relación entre las tres abstracciones

Aunque se presenten como estructuras distintas, internamente se relacionan de la siguiente forma: 

Dataset → DataFrame → RDD → Particiones
  • Un DataFrame es esencialmente un Dataset no tipado (Dataset[Row]).
  • DataFrames y Datasets terminan ejecutándose sobre RDDs.
  • RDD es la base física de ejecución distribuida en Spark Core.

Casos de uso recomendados

Cuándo usar RDD
  • Cuando se necesita control detallado y transformaciones complejas no expresables fácilmente con API SQL.
  • Cuando los datos son no estructurados o el esquema es altamente variable.
  • Cuando se implementan algoritmos muy personalizados con operaciones de bajo nivel.
Cuándo usar DataFrame
  • En procesos ETL y pipelines de transformación.
  • En analítica y consultas sobre datos estructurados.
  • Cuando se busca rendimiento y optimización automática.
  • Para integración con SQL, Parquet, catálogos y sistemas de almacenamiento.
Cuándo usar Dataset
  • En proyectos Scala/Java donde la seguridad de tipos es crítica.
  • En aplicaciones empresariales con modelos de datos definidos (case classes/POJOs).
  • Cuando se quiere combinar tipado fuerte con optimización de consultas.

Implicaciones en rendimiento

De forma general, DataFrame y Dataset suelen superar a RDD en la mayoría de cargas ETL y analíticas debido a:

  • Optimización de planes de ejecución (Catalyst).
  • Pushdown de filtros y selección de columnas (reducción de E/S).
  • Estrategias de joins más eficientes.
  • Mejor uso de memoria y representación interna de datos en ejecución.

RDD puede ser adecuado en casos especiales, pero suele implicar mayor sobrecarga y falta de optimización automática.

Conclusión

RDD, DataFrame y Dataset representan tres niveles de abstracción sobre el mismo motor distribuido de Spark. RDD ofrece control y flexibilidad de bajo nivel, mientras que DataFrame aporta optimización automática y eficiencia para ETL y analítica. Dataset añade, además, seguridad de tipos en lenguajes como Scala y Java.

En desarrollos modernos orientados a procesamiento estructurado, DataFrame suele ser la opción estándar. RDD queda reservado para escenarios específicos donde se requiere un control muy detallado o lógica no estructurada, mientras que Dataset es especialmente útil en aplicaciones Scala/Java que demandan tipado fuerte.

EJERCICIO PROPUESTO

Comparar el rendimiento y el nivel de expresividad entre RDD y DataFrame para una operación equivalente sobre un volumen significativo de datos.

Pasos
  1. Crear un RDD con 1 millón de registros numéricos (por ejemplo, del 1 al 1.000.000).
  2. Aplicar una transformación: multiplicar por 2, filtrar múltiplos de 3 y contar resultados.
  3. Medir tiempo de ejecución aproximado (por ejemplo, con registros de logs o medición simple de tiempo).
  4. Convertir el RDD a DataFrame con una columna.
  5. Repetir la misma operación utilizando la API de DataFrame (expresiones/funciones).
  6. Comparar tiempos y discutir diferencias en términos de optimización (Catalyst) y planificación.
Sugerencia de implementación

Ejemplo base (PySpark) para el RDD:

import time

rdd = spark.sparkContext.parallelize(range(1, 1000001))

t0 = time.time()
count_rdd = rdd.map(lambda x: x * 2).filter(lambda x: x % 3 == 0).count()
t1 = time.time()

print("RDD count:", count_rdd, "Tiempo:", t1 - t0)

Ejemplo base (PySpark) para el DataFrame:

from pyspark.sql.functions import col
import time

df = spark.range(1, 1000001).toDF("x")

t0 = time.time()
count_df = df.select((col("x") * 2).alias("y")).filter(col("y") % 3 == 0).count()
t1 = time.time()

print("DF count:", count_df, "Tiempo:", t1 - t0)
