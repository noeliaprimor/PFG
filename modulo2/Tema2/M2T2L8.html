DESCRIPCIÓN

En Apache Spark, el RDD (Resilient Distributed Dataset) es la abstracción fundamental de Spark Core para el procesamiento distribuido. Las operaciones sobre RDD permiten construir pipelines de procesamiento de datos combinando transformaciones y acciones, que se ejecutan sobre colecciones distribuidas particionadas.

Comprender estas operaciones es esencial para dominar el flujo de ejecución, la construcción del DAG (Directed Acyclic Graph), la planificación de stages y el impacto del shuffle en el rendimiento.

El modelo de ejecución de RDD se basa en la evaluación perezosa (lazy evaluation). Esto significa que Spark no ejecuta inmediatamente las transformaciones, sino que construye un plan lógico (DAG) con el linaje de operaciones. La ejecución real se produce cuando se invoca una acción.

De forma general, el flujo es:

  1. Se definen transformaciones (creación de nuevos RDDs).
  2. Spark registra el linaje y construye el DAG.
  3. Una acción dispara la ejecución del plan completo.
  4. El DAG se divide en stages y tasks según dependencias y shuffles.

TRANSFORMACIONES

Las transformaciones son operaciones que devuelven un nuevo RDD y no ejecutan el cómputo inmediatamente. Construyen el linaje que Spark utilizará para ejecutar y recomputar particiones en caso de fallo.

Transformaciones básicas

map()

Aplica una función a cada elemento del RDD (transformación uno a uno).

rdd2 = rdd.map(lambda x: x * 2)
flatMap()

Similar a map, pero cada elemento puede generar cero, uno o múltiples elementos de salida. Es muy común en procesamiento de texto y transformación de estructuras anidadas. 

rdd2 = rdd.flatMap(lambda x: x.split(" "))
filter()

Filtra elementos según una condición booleana.

rdd2 = rdd.filter(lambda x: x > 10)

Transformaciones por clave (clave-valor)

Muchas operaciones en Spark se basan en pares (clave, valor). Estas transformaciones son fundamentales para agregaciones y reestructuración de datos.

reduceByKey()

Agrupa valores por clave y aplica una reducción. Suele ser más eficiente que groupByKey porque realiza combinaciones parciales en cada partición antes del shuffle. 

rdd2 = rdd.reduceByKey(lambda a, b: a + b)
groupByKey()

Agrupa todos los valores por clave, devolviendo una colección por clave. Suele ser menos eficiente porque envía todos los valores por red durante el shuffle. 

rdd2 = rdd.groupByKey()
combineByKey()

Permite agregaciones más avanzadas con control detallado sobre cómo se inicializan acumuladores, cómo se combinan dentro de una partición y cómo se fusionan entre particiones. Es útil para medias, varianzas u otras métricas compuestas.

Transformaciones estructurales

distinct()

Elimina duplicados. Normalmente implica shuffle, ya que los elementos iguales pueden estar distribuidos en diferentes particiones. 

rdd2 = rdd.distinct()
union()

Une dos RDDs (concatenación distribuida).

rdd3 = rdd1.union(rdd2)
intersection()

Devuelve elementos comunes entre dos RDDs. Suele implicar shuffle y puede ser costosa en datasets grandes. 

rdd3 = rdd1.intersection(rdd2)

Transformaciones de particionado

Estas transformaciones modifican cómo se distribuyen los datos en particiones, lo que afecta al paralelismo, a la ejecución y, en ocasiones, al número de ficheros de salida.

repartition()

Redistribuye los datos creando un nuevo número de particiones, normalmente con shuffle completo (coste mayor, balanceo mejor). 

rdd2 = rdd.repartition(4)
coalesce()

Reduce el número de particiones intentando evitar shuffle completo (coste menor), aunque puede producir desequilibrios en la carga. 

rdd2 = rdd.coalesce(2)

ACCIONES

Las acciones disparan la ejecución del DAG. Devuelven resultados al driver o escriben datos en almacenamiento externo. Son el punto en el que Spark materializa el plan de ejecución.

Acciones básicas

collect()

Devuelve todos los elementos al driver.

resultado = rdd.collect()

Advertencia: es peligroso en grandes volúmenes, ya que puede agotar la memoria del driver.

count()

Cuenta el número de elementos.

total = rdd.count()
first()

Devuelve el primer elemento.

elemento = rdd.first()
take(n)

Devuelve los primeros n elementos.

muestra = rdd.take(5)

Acciones de agregación

reduce()

Aplica una función acumulativa para reducir el RDD a un único valor.

suma = rdd.reduce(lambda a, b: a + b)
aggregate()

Permite agregaciones más controladas con un valor inicial y funciones separadas para combinar dentro de particiones y entre particiones. Resulta útil cuando el tipo acumulador difiere del tipo de los elementos o cuando se requiere lógica más compleja.

Acciones de escritura

saveAsTextFile()

Guarda el contenido del RDD en un directorio de salida. Spark crea múltiples ficheros (uno por partición). 

rdd.saveAsTextFile("salida")

CONSIDERACIONES GENERALES

Dependencias: Narrow vs Wide

Las dependencias entre RDDs determinan cómo se divide el DAG en stages y si se requiere shuffle. Se distinguen dos tipos:

1 Narrow dependencies (dependencias estrechas)

Cada partición hija depende de una única partición padre. No requieren shuffle y suelen ser más eficientes.

Ejemplos típicos:

  • map
  • filter
  • flatMap
2 Wide dependencies (dependencias amplias)

Cada partición hija puede depender de múltiples particiones padre. Normalmente requieren shuffle, implicando transferencia de datos por red y creación de nuevos stages.

Ejemplos típicos:

  • groupByKey
  • reduceByKey
  • distinct
  • intersection (en muchos casos)

Persistencia y cache

Dado que las transformaciones son perezosas, si un RDD se reutiliza en varias acciones, Spark podría recomputarlo desde el linaje cada vez. Para evitarlo, es posible persistirlo:

cache() 
rdd.cache()

Indica a Spark que intente mantener el RDD en memoria (equivalente a persistencia en memoria por defecto).

persist() con niveles de almacenamiento 
from pyspark import StorageLevel
rdd.persist(StorageLevel.MEMORY_AND_DISK)

Permite controlar dónde se almacena (memoria, disco, serializado, etc.). Esto es importante cuando el dataset no cabe en memoria.

Ejemplo completo de pipeline con RDD

El siguiente ejemplo implementa un conteo de palabras (WordCount), un clásico para ilustrar transformaciones y acciones: 

rdd = spark.sparkContext.textFile("texto.txt")

resultado = (rdd
    .flatMap(lambda line: line.split(" "))
    .map(lambda word: (word, 1))
    .reduceByKey(lambda a, b: a + b)
)

print(resultado.take(10))

Flujo conceptual:

  1. Lectura del fichero como RDD de líneas.
  2. Tokenización (flatMap) para generar palabras.
  3. Mapeo a pares (palabra, 1).
  4. Agregación por clave (reduceByKey).
  5. Acción (take) para obtener una muestra y ejecutar el pipeline.

Rendimiento y buenas prácticas

  • Preferir reduceByKey() frente a groupByKey() para agregaciones (menos coste de shuffle).
  • Minimizar operaciones que impliquen shuffle (joins, distinct, groupByKey, repartition).
  • Ajustar particiones según el paralelismo disponible y el tamaño de datos.
  • Evitar collect() en grandes datasets; usar take() o escrituras a almacenamiento.
  • Usar cache() o persist() si se reutiliza el mismo RDD en varias acciones.

CONCLUSIÓN

Las operaciones sobre RDD constituyen la base del modelo distribuido en Spark. Las transformaciones construyen el DAG de ejecución de forma perezosa, mientras que las acciones disparan la ejecución y materializan los resultados. El tipo de dependencias (narrow o wide) determina el número de stages y la necesidad de shuffle, influyendo directamente en el rendimiento.

Aunque DataFrame sea la abstracción más habitual en analítica moderna, entender las operaciones RDD resulta fundamental para comprender el funcionamiento interno de Spark Core, depurar pipelines complejos y optimizar cargas específicas.
