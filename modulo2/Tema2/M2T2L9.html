INTRODUCCIÓN

En Apache Spark, los DataFrames y Datasets constituyen la capa de abstracción estructurada para el procesamiento distribuido. A diferencia de los RDD, estas estructuras incorporan un modelo declarativo (similar a SQL) y se benefician de un conjunto de optimizaciones automáticas que mejoran el rendimiento y reducen el consumo de recursos.

Las principales mejoras frente a RDD provienen de:

  • Optimización automática de consultas mediante el optimizador Catalyst.
  • Planificación lógica y física avanzada (selección de estrategias de ejecución).
  • Gestión eficiente de memoria y ejecución (motor Tungsten y generación de código).
  • Integración nativa con SQL, fuentes de datos y formatos columnar (Parquet/ORC).

Cuando se ejecuta una operación sobre un DataFrame o Dataset, Spark construye un plan de ejecución en varias fases. De forma simplificada:

  1. Se genera un Logical Plan a partir de las operaciones declaradas por el usuario.
  2. Catalyst aplica reglas de optimización y produce un Optimized Logical Plan.
  3. El optimizador elige una estrategia y construye un Physical Plan (operadores concretos de ejecución).
  4. El plan físico se ejecuta sobre el clúster (en última instancia, sobre tareas distribuidas).

Este proceso es transparente para el desarrollador y constituye la base de la mejora de rendimiento respecto a implementaciones equivalentes con RDD.

DATAFRAMES

Operaciones fundamentales sobre DataFrames

Selección y proyección

La selección de columnas permite proyectar únicamente los campos necesarios. Esto influye directamente en el volumen de datos leídos y transferidos. 

df.select("nombre", "edad")
df.selectExpr("nombre", "edad * 2 as edad_doble")

Optimización habitual: column pruning (solo se leen y procesan las columnas utilizadas).

Filtrado

Los filtros pueden expresarse con API de DataFrame o como expresiones SQL: 

df.filter(df.edad > 30)
df.where("edad > 30")

Optimizaciones habituales:

  • Predicate pushdown (cuando el origen lo soporta): empujar filtros al datasource para leer menos datos.
  • Aplicación temprana de filtros para reducir volumen antes de shuffles y joins.
Agregaciones

Las agregaciones se realizan con groupBy y funciones de agregación: 

from pyspark.sql.functions import count, avg

df.groupBy("departamento").agg(count("*"), avg("salario"))

En agregaciones, Spark suele aplicar combinaciones parciales por partición (similar a un combiner) antes de redistribuir datos, reduciendo el coste de red.

Ordenación y ventanas (conceptos frecuentes)

Operaciones como orderBy o funciones de ventana pueden implicar shuffle y ordenación distribuida: 

df.orderBy("salario")

from pyspark.sql.window import Window
from pyspark.sql.functions import row_number

w = Window.partitionBy("departamento").orderBy("salario")
df.withColumn("rn", row_number().over(w))

Estas operaciones suelen ser más costosas que filtros y proyecciones, por lo que conviene aplicarlas después de reducir el dataset.

Join

Un join combina filas de dos DataFrames basándose en una clave: 

df1.join(df2, "id")

Spark selecciona estrategias de join según tamaños estimados, estadísticas y configuración: broadcast join, sort-merge join o shuffle hash join, entre otras.

Tipos de join y optimización

Broadcast join

Si uno de los datasets es pequeño, puede distribuirse (broadcast) a todos los executors para evitar un shuffle grande: 

from pyspark.sql.functions import broadcast

df1.join(broadcast(df2), "id")

Ventajas típicas:

  • Evita redistribución masiva de datos.
  • Reduce coste de red.
  • Muy eficiente en joins asimétricos (tabla grande + tabla pequeña).
Sort-merge join

Estrategia habitual cuando ambos datasets son grandes. Requiere redistribución por clave y ordenación, por lo que suele implicar shuffle y coste elevado, aunque es robusta para grandes volúmenes.

Shuffle hash join

Puede aplicarse cuando las condiciones y tamaños relativos lo hacen adecuado. Implica shuffle, pero la ejecución puede ser eficiente en algunos escenarios según configuración y cardinalidad.

DATASET (SCALA/JAVA) Y OPERACIONES TIPADAS

Un Dataset es una abstracción disponible principalmente en Scala y Java que combina:

  • Optimización Catalyst (como DataFrame).
  • Seguridad de tipos (tipado fuerte) a nivel de compilación.

En términos prácticos, un DataFrame puede considerarse un Dataset no tipado (Dataset[Row]). En PySpark no existe el Dataset tipado como entidad separada; normalmente se trabaja con DataFrames.

Ejemplo en Scala:

case class Persona(nombre: String, edad: Int)
val ds = spark.read.json("personas.json").as[Persona]

Ventajas del enfoque Dataset en Scala/Java:

  • Errores detectados en compilación (por ejemplo, nombres y tipos de campos).
  • Mayor mantenibilidad y refactorización segura en proyectos grandes.
  • Conserva optimización y rendimiento del motor SQL.

CONSIDERACIONES GENERALES

Optimización automática: Catalyst

Catalyst es el optimizador de Spark SQL que aplica reglas sobre el plan lógico para mejorar la ejecución. Entre las optimizaciones habituales se incluyen:

  • Constant folding: evaluación de expresiones constantes en tiempo de planificación.
  • Predicate pushdown: empujar filtros al origen si el conector lo permite.
  • Projection pruning: eliminar columnas no necesarias.
  • Join reordering: reordenamiento de joins cuando existen múltiples joins y estadísticas disponibles.
  • Simplificación y reescritura de expresiones para reducir trabajo.

Ejemplo de pipeline donde Catalyst puede reordenar y simplificar:

df.filter(df.edad > 30).select("nombre")

Spark intentará aplicar el filtro y la proyección en el orden más eficiente, y eliminará columnas innecesarias lo antes posible.

Tungsten y ejecución eficiente

Tungsten se refiere a un conjunto de mejoras orientadas a la eficiencia de ejecución y memoria, destacando:

  • Gestión eficiente de memoria para reducir overhead de objetos y mejorar locality.
  • Representación binaria más compacta en ejecución.
  • Whole-stage code generation: generación de código para encadenar operadores y reducir overhead.

Estas optimizaciones suelen traducirse en mejoras significativas respecto a aproximaciones equivalentes con RDD, especialmente en cargas ETL/analítica.

Análisis del plan de ejecución

Spark permite inspeccionar cómo se ejecutará una operación mediante explain(). Esto es clave para analizar joins, shuffles y decisiones del optimizador: 

df.explain()

Para obtener información más detallada:

df.explain(True)

En los planes se pueden observar, entre otros:

  • Logical plan y optimized logical plan.
  • Physical plan (operadores concretos).
  • Estrategia de join seleccionada.
  • Indicadores de shuffle (Exchange) y ordenación (Sort).

Particionado, shuffle y optimización

La distribución de datos en particiones influye directamente en el rendimiento. En DataFrames, operaciones como joins y groupBy suelen provocar shuffles, por lo que ajustar el particionado es una palanca importante.

Repartition y coalesce 
df_repart = df.repartition(10)
df_coal = df.coalesce(2)

Recomendaciones generales:

  • Evitar repartition innecesario (puede provocar shuffle costoso).
  • Usar coalesce para reducir particiones antes de escribir y evitar demasiados ficheros.
  • Ajustar spark.sql.shuffle.partitions según el tamaño de datos y el clúster.
Configuración típica de particiones de shuffle 
spark.conf.set("spark.sql.shuffle.partitions", 200)

Este parámetro controla el número de particiones utilizadas por defecto tras operaciones que implican shuffle en Spark SQL.

Caching y persistencia

Si un DataFrame se reutiliza en múltiples acciones, cachear puede evitar recomputación: 

df.cache()

Alternativamente, se puede usar persistencia con niveles específicos (según API disponible en el lenguaje): 

df.persist()

Esto resulta especialmente útil cuando el plan es costoso (joins, agregaciones) y el dataset intermedio se reutiliza varias veces.

UDF vs funciones nativas

Las UDF (User Defined Functions) permiten lógica personalizada, pero pueden perjudicar el rendimiento porque limitan la capacidad de Catalyst para optimizar expresiones: 

from pyspark.sql.functions import udf

Recomendación general:

  • Preferir funciones nativas de pyspark.sql.functions siempre que sea posible.
  • Evitar UDF en transformaciones críticas; considerar SQL expressions, funciones integradas o UDFs vectorizadas (según caso).

Buenas prácticas de optimización

  • Usar DataFrames/SQL para ETL y analítica: favorece optimización automática.
  • Aplicar filtros y proyecciones lo antes posible para reducir volumen.
  • Minimizar shuffles: revisar groupBy, joins, distinct y repartition.
  • Usar broadcast join cuando exista una tabla pequeña en joins asimétricos.
  • Ajustar spark.sql.shuffle.partitions a la escala real del dataset.
  • Revisar planes con explain(True) para validar estrategia y detectar cuellos de botella.
  • Cachear datasets intermedios reutilizados.

CONCLUSIÓN

Las operaciones sobre DataFrames y Datasets representan la forma moderna y optimizada de trabajar en Spark. La combinación de planificación lógica y física, optimización automática mediante Catalyst y mejoras de ejecución (Tungsten y generación de código) permiten obtener mayor rendimiento y escalabilidad que aproximaciones equivalentes basadas en RDD para datos estructurados.

En aplicaciones actuales orientadas a ETL y analítica, DataFrame/Dataset se considera el estándar recomendado, y el uso de herramientas como explain(), broadcast join, ajuste de particiones y caching resulta clave para optimizar costes y tiempos de ejecución.

1. Usa DataFrames/Datasets en lugar de RDDs para aprovechar las optimizaciones de Catalyst.

2. Evita operaciones que desencadenen shuffle innecesarios, como groupByKey.

3. Usa persistencia/caching para conjuntos de datos reutilizados.

4. Aplica broadcast joins para tablas pequeñas.
