Spark Streaming y Structured Streaming

INTRODUCCIÓN

El procesamiento de datos en tiempo real se ha convertido en un componente esencial en arquitecturas modernas de Big Data. Apache Spark incorpora capacidades de streaming mediante dos enfoques históricos:

  • Spark Streaming (basado en DStreams).
  • Structured Streaming (basado en DataFrames/Datasets).

Aunque ambos permiten procesar datos en flujo continuo, Structured Streaming representa la evolución del modelo hacia un enfoque declarativo, optimizado e integrado con Spark SQL.

CONTENIDO

Spark Streaming

Spark Streaming fue la primera solución de streaming en Spark. Se basa en la abstracción denominada DStream (Discretized Stream), que representa una secuencia de RDDs generados a intervalos regulares.

El modelo operativo se basa en micro-batches: los datos que llegan en un intervalo de tiempo se agrupan y procesan como un pequeño lote (batch).

Arquitectura conceptual
  • Recepción de datos (por ejemplo, Kafka, socket, archivos).
  • División en micro-batches (intervalo configurable).
  • Procesamiento distribuido mediante RDD.
  • Salida a almacenamiento o sistema externo.
Ejemplo básico (Scala) 
import org.apache.spark.streaming._
import org.apache.spark.streaming.socket._

val ssc = new StreamingContext(sc, Seconds(5))
val lines = ssc.socketTextStream("localhost", 9999)
val words = lines.flatMap(_.split(" "))
val pairs = words.map(word => (word, 1))
val counts = pairs.reduceByKey(_ + _)

counts.print()
ssc.start()
ssc.awaitTermination()
Limitaciones de Spark Streaming
  • Basado en RDD, menor integración con Spark SQL.
  • API menos declarativa y más compleja.
  • Gestión de estado menos flexible.
  • Mayor complejidad en optimización y mantenimiento.

Debido a estas limitaciones, Structured Streaming se convirtió en la solución recomendada en versiones modernas de Spark.

Structured Streaming

Structured Streaming es la evolución de Spark Streaming. Se basa en el mismo modelo estructurado que Spark SQL y DataFrames, permitiendo escribir consultas declarativas que se ejecutan de forma incremental sobre datos en streaming.

El desarrollador define una consulta sobre un DataFrame de streaming, y Spark se encarga de:

  • Gestionar micro-batches o ejecución continua.
  • Mantener el estado cuando sea necesario.
  • Aplicar optimización automática (Catalyst).
Modelo de programación

Structured Streaming trata el flujo de datos como una tabla infinita que crece continuamente. Cada operación sobre esta “tabla” se ejecuta incrementalmente.

Ejemplo básico (PySpark con Kafka) 
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("StreamingExample").getOrCreate()

stream_df = (spark
    .readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "localhost:9092")
    .option("subscribe", "topic")
    .load())

query = (stream_df
    .writeStream
    .format("console")
    .start())

query.awaitTermination()

Modos de salida en Structured Streaming

Structured Streaming soporta diferentes modos de salida:

  • Append: solo nuevas filas añadidas.
  • Update: filas actualizadas.
  • Complete: reescribe todo el resultado en cada trigger.

Ejemplo con agregación:

from pyspark.sql.functions import count

agg_df = stream_df.groupBy("key").agg(count("*").alias("total"))

query = (agg_df
    .writeStream
    .outputMode("complete")
    .format("console")
    .start())

Ventanas temporales y gestión de estado

Ventanas por tiempo 
from pyspark.sql.functions import window

windowed = (stream_df
    .groupBy(window(stream_df.timestamp, "10 minutes"))
    .count())
Watermarking

Permite gestionar datos tardíos (late data) definiendo un umbral temporal para aceptar eventos atrasados. 

stream_df.withWatermark("timestamp", "5 minutes")

Arquitectura y ejecución

Structured Streaming sigue utilizando micro-batches en la mayoría de configuraciones, aunque puede trabajar en modo continuo según el origen y la configuración.

Cada trigger:

  1. Lee nuevos datos desde la fuente.
  2. Aplica transformaciones incrementales.
  3. Actualiza estado si es necesario.
  4. Escribe resultados en el sink configurado.

Casos de uso típicos

  • Procesamiento de logs en tiempo real.
  • Detección de fraude en transacciones.
  • Monitoreo de sistemas.
  • Agregaciones en ventanas temporales.
  • Ingesta continua hacia data lakes.

CONCLUSIÓN

Spark Streaming introdujo el procesamiento distribuido en tiempo real dentro del ecosistema Spark mediante micro-batches basados en RDD. Structured Streaming representa su evolución natural, ofreciendo una API declarativa, optimización automática y plena integración con Spark SQL y DataFrames.

EJERCICIO RESUELTO PASO A PASO

Procesamiento de valoraciones de películas en tiempo real

En este ejercicio se trabaja con Structured Streaming simulando un caso sencillo: una plataforma de cine recibe valoraciones de películas de forma continua.

A diferencia de los ejemplos que utilizan la fuente rate, en este caso los eventos no se generan automáticamente a partir de un contador. Los eventos proceden de ficheros CSV que se van incorporando poco a poco a un directorio de entrada.

Esta aproximación resulta más intuitiva para entender el concepto de streaming: Spark permanece escuchando un directorio y procesa automáticamente los nuevos ficheros que aparecen en él.

Escenario del ejercicio

Se parte de una carpeta llamada entrada_valoraciones. Cada vez que se copie un nuevo fichero CSV en esa carpeta, Spark lo detectará y procesará sus registros como nuevos eventos.

Un fichero de entrada podría tener el siguiente contenido:

pelicula,usuario,valoracion
Matrix,u001,5
Interstellar,u002,4
Inception,u003,3
Matrix,u004,4

Cada fila representa una valoración realizada por un usuario. El objetivo será calcular, de forma continua, cuántas valoraciones recibe cada película y cuál es su valoración media.

Pasos del ejercicio

  1. Crear una sesión de Spark.
  2. Definir el esquema de los datos de entrada.
  3. Leer los ficheros CSV como un flujo continuo.
  4. Agrupar los datos por película.
  5. Calcular el número de valoraciones y la valoración media.
  6. Mostrar los resultados por consola.
