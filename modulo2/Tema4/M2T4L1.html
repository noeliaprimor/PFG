Capítulo X: Monitorización y configuración de Apache Spark

1. Introducción

La monitorización y correcta configuración de Apache Spark son aspectos fundamentales para garantizar rendimiento, estabilidad y eficiencia en entornos de procesamiento distribuido. Una mala configuración puede provocar cuellos de botella, consumo excesivo de memoria, tiempos elevados de ejecución o incluso fallos de aplicación.

En entornos académicos y profesionales, comprender cómo ajustar parámetros y cómo interpretar métricas de ejecución resulta clave para optimizar aplicaciones Spark.

2. Arquitectura relevante para la monitorización

Spark ejecuta aplicaciones mediante:

  • Driver: coordina la ejecución.
  • Executors: ejecutan tareas en los nodos del clúster.
  • Tasks: unidades de trabajo dentro de un stage.
  • Stages: agrupaciones de tareas delimitadas por operaciones de shuffle.

La monitorización se centra en observar el comportamiento de estos componentes durante la ejecución.

3. Spark Web UI

3.1 Descripción

Spark proporciona una interfaz web integrada (Spark UI) accesible generalmente en el puerto 4040 durante la ejecución de una aplicación.

3.2 Secciones principales

  • Jobs: muestra trabajos ejecutados y su estado.
  • Stages: detalla etapas, tareas, tiempos y shuffles.
  • Storage: muestra datasets cacheados y uso de memoria.
  • Environment: configuración activa de Spark.
  • Executors: uso de CPU, memoria y número de tareas por executor.
  • SQL: planes de ejecución y métricas de consultas SQL.

3.3 Métricas importantes

  • Tiempo total por stage.
  • Shuffle read/write.
  • Skew (desequilibrio entre tareas).
  • Uso de memoria por executor.
  • GC time (tiempo de recolección de basura).

4. Logs y eventos

4.1 Logs de aplicación

Spark utiliza sistemas de logging (por ejemplo, Log4j) para registrar eventos de ejecución.

4.2 Event Logs

Puede activarse el historial de aplicaciones: 

spark.eventLog.enabled=true
spark.eventLog.dir=/ruta/a/logs

Posteriormente, estos logs pueden visualizarse mediante el Spark History Server.

5. Configuración de Spark

5.1 Formas de configuración

  • Parámetros en spark-submit.
  • Archivo spark-defaults.conf.
  • Configuración programática en SparkSession.builder.

5.2 Ejemplo programático

from pyspark.sql import SparkSession

spark = (SparkSession.builder
    .appName("MiApp")
    .config("spark.executor.memory", "4g")
    .config("spark.executor.cores", "2")
    .config("spark.sql.shuffle.partitions", "200")
    .getOrCreate())

6. Parámetros clave de configuración

6.1 Memoria

  • spark.executor.memory: memoria por executor.
  • spark.driver.memory: memoria del driver.
  • spark.memory.fraction: fracción para ejecución y almacenamiento.

6.2 Paralelismo

  • spark.executor.cores: núcleos por executor.
  • spark.default.parallelism: número por defecto de particiones.
  • spark.sql.shuffle.partitions: particiones tras operaciones de shuffle.

6.3 Adaptive Query Execution (AQE)

spark.sql.adaptive.enabled=true
spark.sql.adaptive.skewJoin.enabled=true

AQE permite optimizar dinámicamente joins y particiones durante la ejecución.

7. Optimización del rendimiento

7.1 Gestión de particiones

df.repartition(200)
df.coalesce(10)

Repartition aumenta el paralelismo; coalesce reduce particiones sin shuffle completo.

7.2 Cache y persistencia

df.cache()
df.persist()

Útil cuando un dataset se reutiliza múltiples veces.

7.3 Broadcast joins

from pyspark.sql.functions import broadcast

df_join = df1.join(broadcast(df2), "id")

8. Monitorización en clúster

8.1 Standalone

Proporciona UI del Master y Workers para visualizar uso de recursos.

8.2 YARN

Integración con ResourceManager y ApplicationMaster.

8.3 Kubernetes

Monitorización mediante herramientas como kubectl, dashboards y métricas del clúster.

9. Buenas prácticas

  • Evitar collect() en datasets grandes.
  • Analizar explain(True) para entender planes físicos.
  • Controlar número de particiones tras shuffle.
  • Monitorizar skew en stages largos.
  • Configurar adecuadamente memoria según volumen de datos.

10. Ejercicio propuesto (nivel TFG)

10.1 Objetivo

Analizar el comportamiento de una aplicación Spark variando parámetros de configuración y observando métricas en la Spark UI.

10.2 Enunciado

  1. Ejecutar una aplicación con configuración por defecto.
  2. Registrar métricas de tiempo, shuffle y uso de memoria.
  3. Modificar spark.sql.shuffle.partitions.
  4. Activar AQE.
  5. Comparar tiempos y planes físicos.
  6. Documentar conclusiones.

11. Conclusión

La monitorización y configuración de Spark son elementos esenciales para obtener un rendimiento óptimo. La comprensión de métricas, configuración de memoria, paralelismo y optimización adaptativa permite ajustar aplicaciones a distintos entornos y cargas de trabajo.

En el contexto de un TFG, este conocimiento demuestra dominio práctico del ecosistema Spark, más allá del uso básico de sus APIs.

SPARK UI

El Spark UI (por defecto en http://[driver]:4040 mientras la app está viva) muestra:

  • Jobs / Stages / Tasks (tiempos, fallos, shuffle)
  • Storage (RDD/DataFrame cached)
  • Environment (configuración de Spark)
  • Executors (uso de CPU/memoria por executor)

Usa el UI para identificar cuellos de botella: tareas lentas, imbalance de particiones o problemas de GC.

SISTEMA DE MÉTRICAS: Prometheus + Grafana

Spark puede exportar métricas via Dropwizard -> Prometheus collector. Con Grafana puedes crear dashboards de latencia, throughput y utilización.

# ejemplo de configuración en spark-defaults.conf
spark.metrics.conf.*.sink.prometheusServlet.class=org.apache.spark.metrics.sink.PrometheusServlet
spark.metrics.conf.*.sink.prometheusServlet.path=/metrics

CONFIGURACIÓN CLAVE

  • spark.driver.memory y spark.executor.memory — para ajustar memoria del driver y executors.
  • spark.executor.cores — núcleos por executor.
  • spark.sql.shuffle.partitions — particiones por defecto tras operaciones de shuffle.
  • spark.memory.fraction / spark.memory.storageFraction — control de memoria para ejecución vs cache.

Ejemplo: para un cluster con 8 cores por nodo y 32GB RAM, una configuración típica por executor: --executor-memory 8G --executor-cores 4, y ajustar parallelism a un múltiplo de los cores totales.
