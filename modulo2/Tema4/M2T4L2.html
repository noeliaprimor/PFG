Capítulo X: Seguridad y despliegue en Apache Spark

Introducción

En entornos Big Data, la seguridad y el despliegue adecuado de la infraestructura son aspectos críticos. Apache Spark, como motor distribuido de procesamiento, suele ejecutarse en clústeres compartidos y procesar datos sensibles (financieros, personales o estratégicos), lo que exige mecanismos robustos de autenticación, autorización, cifrado y aislamiento.

Este capítulo aborda los principales mecanismos de seguridad disponibles en Spark y los modelos habituales de despliegue en entornos productivos.

Modelos de despliegue

Modo Local

Utilizado para desarrollo y pruebas. Spark se ejecuta en una única máquina utilizando múltiples hilos. No es adecuado para producción.

Modo Standalone

Spark incluye su propio gestor de clúster (Spark Master y Workers). Es sencillo de configurar y apropiado para entornos controlados o académicos.

YARN

Permite ejecutar Spark sobre el ecosistema Hadoop, integrándose con el ResourceManager. Es habitual en infraestructuras empresariales tradicionales.

Kubernetes

Permite desplegar Spark como contenedores gestionados por Kubernetes. Es el modelo preferido en arquitecturas cloud-native modernas.

Seguridad en Apache Spark

Autenticación

Spark puede configurarse para requerir autenticación entre el driver y los executors: 

spark.authenticate=true
spark.authenticate.secret=clave_secreta

En entornos Hadoop, suele integrarse con Kerberos para autenticación fuerte.

Autorización

Spark permite controlar acceso a la Spark UI mediante listas de control: 

spark.acls.enable=true
spark.ui.view.acls=usuario1,usuario2
spark.modify.acls=admin

En entornos empresariales, la autorización suele gestionarse a nivel de plataforma (por ejemplo, mediante YARN, Kubernetes o soluciones externas).

Cifrado en tránsito (TLS/SSL)

Para proteger comunicaciones entre nodos, puede habilitarse cifrado: 

spark.ssl.enabled=true
spark.ssl.keyStore=/ruta/keystore
spark.ssl.keyStorePassword=********

Esto protege datos intercambiados entre driver, executors y servicios auxiliares.

Cifrado en reposo

El cifrado en reposo depende del sistema de almacenamiento subyacente (HDFS, S3, discos locales, etc.).

Spark puede cifrar datos temporales (shuffle y spill): 

spark.io.encryption.enabled=true

Aislamiento y multi-tenancy

En clústeres compartidos, múltiples usuarios pueden ejecutar aplicaciones simultáneamente. Para evitar interferencias:

  • Definir límites de CPU y memoria por aplicación.
  • Utilizar colas en YARN.
  • Aplicar namespaces y cuotas en Kubernetes.
  • Separar entornos (desarrollo, preproducción, producción).

Seguridad de datos sensibles

  • Evitar imprimir datos sensibles en logs.
  • Controlar permisos de lectura/escritura en almacenamiento.
  • Usar variables de entorno seguras para credenciales.
  • No hardcodear claves en el código.

Despliegue con spark-submit

Ejemplo en modo Standalone 
spark-submit \
  --master spark://master:7077 \
  --deploy-mode cluster \
  --executor-memory 4g \
  --executor-cores 2 \
  app.py
Ejemplo en Kubernetes 
spark-submit \
  --master k8s://https://k8s-api:6443 \
  --deploy-mode cluster \
  --conf spark.executor.instances=3 \
  --conf spark.kubernetes.container.image=spark-image \
  app.py

Gestión de dependencias

  • Uso de paquetes mediante --packages.
  • Incluir dependencias en un JAR (Scala/Java).
  • Uso de entornos virtuales en PySpark.
  • Construcción de imágenes Docker en entornos Kubernetes.

Buenas prácticas de despliegue

  • Versionar configuraciones.
  • Separar configuración de código.
  • Automatizar despliegues (CI/CD).
  • Monitorizar recursos tras despliegue.
  • Probar cargas antes de producción.

Alta disponibilidad

Master en modo HA (Standalone)

Puede configurarse más de un master utilizando ZooKeeper para coordinación.

Tolerancia a fallos
  • Reintento automático de tareas.
  • Recomputación basada en lineage.
  • Reinicio automático de pods en Kubernetes.

Ejercicio propuesto (nivel TFG)

Objetivo

Diseñar una arquitectura segura de despliegue Spark para un entorno empresarial que procese datos sensibles.

Enunciado
  1. Seleccionar modelo de despliegue (Standalone, YARN o Kubernetes).
  2. Definir mecanismos de autenticación y cifrado.
  3. Especificar parámetros de aislamiento de recursos.
  4. Describir estrategia de alta disponibilidad.
  5. Identificar riesgos y medidas mitigadoras.

Kerberos

Integración de Spark con Kerberos para autenticación: configurar principal y keytab, y arrancar procesos con kinit.

TLS / SSL

Habilitar TLS para comunicaciones entre driver y executors y para el REST API si es necesario.

# ejemplo: arrancar con keytab
kinit -kt /etc/security/keytabs/spark.service.keytab spark/hostname@REALM
spark-submit --principal user@REALM --keytab /path/to/user.keytab ...
CONCLUSIÓN

La seguridad y el despliegue adecuado de Spark son fundamentales para entornos productivos. No basta con desarrollar aplicaciones eficientes; es necesario proteger datos, asegurar comunicaciones y garantizar disponibilidad.

La correcta configuración de autenticación, cifrado, aislamiento de recursos y alta disponibilidad permite construir plataformas de datos robustas y alineadas con requisitos empresariales y normativos.
