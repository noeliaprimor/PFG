Seguridad y despliegue

INTRODUCCIÓN

En entornos Big Data, la seguridad y el despliegue adecuado de la infraestructura son aspectos críticos. Apache Spark, como motor distribuido de procesamiento, suele ejecutarse en clústeres compartidos y procesar datos sensibles (financieros, personales o estratégicos), lo que exige mecanismos robustos de autenticación, autorización, cifrado y aislamiento.

Esta lección aborda los principales mecanismos de seguridad disponibles en Spark que se apoya en varios pilares fundamentales:

  • Autenticación.
  • Autorización.
  • Cifrado de comunicaciones.
  • Aislamiento de procesos.
  • Control de acceso a interfaces web.
  • Gestión segura de credenciales.

CONTENIDO

Autenticación

La autenticación permite verificar la identidad de los componentes que participan en una aplicación Spark, como el Driver, los Executors y el gestor del clúster.

Autenticación mediante secreto compartido

Spark permite activar autenticación básica mediante un secreto compartido entre los componentes del sistema.

spark.authenticate=true
spark.authenticate.secret=mi_secreto

Este mecanismo ayuda a evitar que procesos no autorizados se conecten a la aplicación Spark.

Integración con Kerberos

En entornos Hadoop empresariales, Spark puede integrarse con Kerberos para proporcionar autenticación fuerte basada en tickets.

kinit usuario@REALM

spark-submit \
  --principal usuario@REALM \
  --keytab usuario.keytab \
  app.py

Kerberos resulta especialmente relevante cuando Spark accede a servicios protegidos como HDFS, Hive o YARN.

Cifrado de comunicaciones

En una aplicación Spark distribuida, los datos circulan entre distintos componentes del sistema. Por ello, es importante proteger la comunicación entre Driver, Executors y otros servicios.

Spark permite activar cifrado de red mediante propiedades de configuración:

spark.network.crypto.enabled=true
spark.ssl.enabled=true

El cifrado ayuda a proteger los datos en tránsito y reduce el riesgo de exposición de información sensible dentro de la red del clúster.

Seguridad en Spark UI

Spark UI proporciona una interfaz web para consultar información detallada sobre la ejecución de una aplicación. Aunque resulta muy útil para depuración y monitorización, también puede exponer información sensible si no se protege adecuadamente.

Entre los riesgos asociados a Spark UI se encuentran:

  • Exposición de nombres de aplicaciones.
  • Visualización de rutas de ficheros.
  • Acceso a métricas internas.
  • Consulta de errores y trazas de ejecución.

Para limitar el acceso pueden activarse listas de control de acceso:

spark.ui.acls.enable=true
spark.ui.view.acls=usuario1,usuario2

En entornos productivos también es recomendable proteger Spark UI mediante redes privadas, proxies autenticados o reglas de firewall.

Control de acceso a datos

Spark no implementa por sí mismo un sistema completo de gobierno y autorización sobre los datos. En la práctica, el control de acceso depende del sistema de almacenamiento o de herramientas externas.

Algunos mecanismos habituales son:

  • Permisos de HDFS.
  • Políticas de acceso en sistemas cloud como Amazon S3, Azure Data Lake o Google Cloud Storage.
  • Herramientas de gobierno de datos como Apache Ranger o Apache Sentry.
  • Catálogos de metadatos con control de permisos.

De esta forma, Spark actúa como motor de procesamiento, mientras que la autorización sobre los datos se gestiona en las capas de almacenamiento y gobierno.

Seguridad en entornos Docker y Kubernetes

Es habitual desplegar Spark mediante Docker, ya que permite crear un entorno reproducible y sencillo de ejecutar. Sin embargo, incluso en este contexto deben considerarse ciertos aspectos de seguridad.

Seguridad en Docker

Algunas buenas prácticas en entornos Docker son:

  • Utilizar redes privadas entre contenedores.
  • No exponer puertos innecesarios al exterior.
  • Evitar incluir credenciales directamente en las imágenes.
  • Usar variables de entorno o ficheros externos para configuración sensible.
  • Mantener las imágenes actualizadas.
  • Ejecutar los contenedores con los mínimos privilegios necesarios.
Seguridad en Kubernetes

En Kubernetes, Spark puede beneficiarse de mecanismos adicionales de seguridad y aislamiento.

  • Namespaces: permiten separar entornos o aplicaciones.
  • RBAC: controla qué usuarios o servicios pueden realizar acciones sobre el clúster.
  • Secrets: permiten gestionar credenciales de forma más segura.
  • Resource limits: evitan que una aplicación consuma más recursos de los asignados.
  • Network policies: restringen la comunicación entre pods.

Despliegue de aplicaciones Spark

La herramienta principal para desplegar aplicaciones Spark es spark-submit. Esta utilidad permite enviar una aplicación al entorno de ejecución indicando el modo de despliegue, recursos necesarios y dependencias.

spark-submit \
  --master spark://spark-master:7077 \
  --deploy-mode client \
  app.py
Modo client

En el modo client, el Driver se ejecuta en la máquina desde la que se lanza la aplicación. Este modo facilita la depuración, ya que los logs del Driver aparecen directamente en la consola del usuario.

Modo cluster

En el modo cluster, el Driver se ejecuta dentro del clúster. Este enfoque es más adecuado para entornos productivos, ya que no depende de la máquina cliente una vez enviada la aplicación.

Gestión de dependencias

Una parte importante del despliegue consiste en garantizar que todas las dependencias necesarias estén disponibles en tiempo de ejecución.

En Spark pueden incluirse dependencias mediante diferentes opciones:

  • --jars para añadir ficheros JAR.
  • --packages para descargar paquetes Maven.
  • --py-files para distribuir código Python adicional.
  • Imágenes Docker personalizadas en entornos contenerizados.
spark-submit \
  --packages org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.0 \
  app.py

Una gestión incorrecta de dependencias puede provocar errores de compatibilidad, fallos en tiempo de ejecución o comportamientos inesperados.

Monitorización y observabilidad

El despliegue de aplicaciones Spark en entornos reales requiere mecanismos de monitorización y observabilidad. Estos permiten conocer el estado de las aplicaciones, analizar su rendimiento y detectar errores.

Algunas herramientas habituales son:

  • Spark UI: permite visualizar jobs, stages, tasks y métricas de ejecución.
  • History Server: permite consultar ejecuciones anteriores a partir de event logs.
  • Prometheus: permite recolectar métricas.
  • Grafana: permite construir dashboards de monitorización.
  • ELK: permite centralizar y analizar logs.

Buenas prácticas de seguridad y despliegue

Buenas prácticas de seguridad
  • Activar autenticación en entornos distribuidos.
  • Usar cifrado de red cuando se transmitan datos sensibles.
  • Limitar el acceso a Spark UI.
  • No almacenar credenciales en código fuente.
  • Gestionar permisos en la capa de almacenamiento.
  • Revisar logs para detectar accesos o errores anómalos.
Buenas prácticas de despliegue
  • Separar entornos de desarrollo, pruebas y producción.
  • Automatizar despliegues siempre que sea posible.
  • Definir recursos adecuados para Driver y Executors.
  • Controlar versiones de Spark, Python, Java y librerías externas.
  • Configurar correctamente logs y event logs.
  • Probar las aplicaciones con volúmenes de datos representativos.

CONCLUSIÓN

La seguridad y el despliegue son aspectos esenciales en cualquier arquitectura basada en Apache Spark. Aunque durante el desarrollo académico pueden simplificarse, en entornos profesionales resultan necesarios para garantizar la protección de los datos, la estabilidad del sistema y la escalabilidad de las aplicaciones.

Comprender los distintos modos de despliegue, los mecanismos de autenticación, el cifrado de comunicaciones, la gestión de dependencias y la monitorización permite diseñar soluciones Spark más robustas, mantenibles y preparadas para escenarios reales de Big Data.

EJERCICIO RESUELTO

Revisión de seguridad de un despliegue Spark

En este ejercicio se revisará la configuración de una aplicación Spark antes de moverla de un entorno de laboratorio a un entorno compartido.

El objetivo es identificar configuraciones inseguras y proponer una versión mejorada del comando spark-submit.

Escenario inicial

Necesitamos una aplicación Spark, vamos a utilizar la de la lección anterior disponible aquí. 

spark-submit \
  --master "local[*]" \
  --conf spark.authenticate=false \
  --conf spark.ui.acls.enable=false \
  --conf spark.eventLog.enabled=false \
  M2T4L1_ejercicio.py

Problemas detectados

  • La autenticación está desactivada.
  • El acceso a Spark UI no está restringido.
  • No se almacenan event logs para revisar ejecuciones anteriores.
  • No se documenta dónde se guardan los eventos de ejecución.

Versión mejorada del despliegue

mkdir -p /tmp/spark-events

spark-submit \
  --master "local[*]" \
  --conf spark.authenticate=true \
  --conf spark.authenticate.secret=secreto_laboratorio \
  --conf spark.ui.acls.enable=true \
  --conf spark.ui.view.acls=noelia \
  --conf spark.eventLog.enabled=true \
  --conf spark.eventLog.dir=file:///tmp/spark-events \
  M2T4L1_ejercicio.py

Comprobaciones a realizar

  1. Ejecutar la aplicación con la configuración mejorada.
  2. Comprobar que la aplicación finaliza correctamente.
  3. Verificar que se han generado ficheros en /tmp/spark-events.
  4. Revisar qué información aparece en Spark UI.
  5. Identificar qué parámetros deberían cambiar antes de pasar a producción.

Resultado esperado

El alumno debe comprender que desplegar una aplicación Spark no consiste únicamente en ejecutarla, sino también en revisar su configuración de seguridad, observabilidad y acceso.

En un entorno real, el secreto no debería escribirse directamente en el comando, el acceso a Spark UI debería estar protegido mediante mecanismos externos y los logs deberían almacenarse en una ubicación persistente y controlada.

Conclusión del ejercicio

Este ejercicio permite distinguir entre una ejecución válida para pruebas locales y una configuración más adecuada para un entorno compartido. La seguridad en Spark debe revisarse antes del despliegue, especialmente cuando se procesan datos sensibles o se trabaja en clústeres utilizados por varios usuarios.
