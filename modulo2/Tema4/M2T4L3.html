Errores frecuentes y cómo solucionarlos

  • Descripción
  • Contenido
  • Conclusión

DESCRIPCIÓN

Durante el desarrollo y despliegue de aplicaciones en Apache Spark es habitual encontrarse con errores relacionados con memoria, particionado, configuración del clúster o uso incorrecto de las APIs. Comprender el origen de estos errores y saber interpretarlos correctamente es esencial para desarrollar aplicaciones robustas y optimizadas.

Esta lección recopila los errores más frecuentes en Spark y describe sus causas habituales junto con estrategias prácticas de resolución.

CONTENIDO

Errores relacionados con memoria

OutOfMemoryError (Java heap space)

Causa:

  • Dataset demasiado grande para la memoria asignada.
  • Uso excesivo de collect() o operaciones que traen datos al driver.
  • Shuffle masivo sin suficiente memoria.

Soluciones:

  • Aumentar spark.executor.memory y/o spark.driver.memory.
  • Evitar collect() en datasets grandes.
  • Revisar número de particiones.
  • Usar persist(StorageLevel.DISK_ONLY) si la memoria es limitada.
GC Overhead Limit Exceeded

Causa: exceso de tiempo en recolección de basura debido a presión de memoria.

Soluciones:

  • Aumentar memoria del executor.
  • Reducir número de objetos intermedios.
  • Optimizar particionado y evitar estructuras complejas innecesarias.

Problemas de particionado y rendimiento

Tareas muy lentas (Data Skew)

Causa: distribución desigual de datos entre particiones.

Soluciones:

  • Activar Adaptive Query Execution (AQE).
  • Reparticionar los datos (repartition()).
  • Aplicar técnicas de salting en claves con alta frecuencia.
Demasiadas particiones pequeñas

Causa: exceso de spark.sql.shuffle.partitions.

Soluciones:

  • Ajustar spark.sql.shuffle.partitions.
  • Usar coalesce() antes de escribir resultados.

Errores de serialización

Task not serializable

Causa: intento de enviar objetos no serializables a los executors.

Soluciones:

  • Evitar capturar objetos complejos dentro de funciones lambda.
  • Marcar clases como Serializable (en Scala).
  • Usar variables broadcast para objetos grandes.
Kryo serialization errors

Causa: tipos no registrados correctamente en Kryo.

Solución: registrar clases personalizadas y habilitar Kryo:

spark.serializer=org.apache.spark.serializer.KryoSerializer

Errores en joins y shuffles

Shuffle fetch failed

Causa: fallo en transferencia de datos entre nodos.

Soluciones:

  • Revisar conectividad de red.
  • Aumentar memoria y timeout.
  • Reducir tamaño de particiones.
Broadcast timeout

Causa: tabla demasiado grande para broadcast.

Soluciones:

  • Aumentar spark.sql.broadcastTimeout.
  • Evitar broadcast manual si la tabla es grande.

Problemas en Structured Streaming

Checkpoint corruption

Causa: interrupción inesperada o almacenamiento inconsistente.

Solución:

  • Eliminar directorio de checkpoint (si procede).
  • Usar almacenamiento fiable (HDFS, S3).
Datos tardíos (late data)

Causa: eventos fuera de la ventana esperada.

Solución: configurar watermark correctamente.

Errores de configuración y despliegue

Executor lost

Causa: falta de recursos o fallo del nodo.

Soluciones:

  • Revisar logs del executor.
  • Aumentar recursos.
  • Verificar configuración del cluster manager.
Cannot allocate memory

Causa: recursos insuficientes en el nodo.

Solución:

  • Ajustar memoria y cores.
  • Reducir paralelismo.

Buenas prácticas para evitar errores

  • Analizar siempre el plan de ejecución con explain(True).
  • Monitorizar Spark UI regularmente.
  • Ajustar particiones según tamaño del dataset.
  • Evitar acciones costosas innecesarias.
  • Separar entorno de desarrollo y producción.

Estrategia sistemática de diagnóstico

  1. Revisar mensaje completo del error.
  2. Analizar logs del driver y executors.
  3. Revisar Spark UI (stages y tareas).
  4. Identificar si el problema es de memoria, red, configuración o datos.
  5. Aplicar cambios graduales y volver a probar.
CONCLUSIÓN

Los errores en Spark suelen estar relacionados con memoria, particionado, serialización o configuración del clúster. Una comprensión profunda del modelo de ejecución y una monitorización adecuada permiten diagnosticar y resolver problemas de forma sistemática.
