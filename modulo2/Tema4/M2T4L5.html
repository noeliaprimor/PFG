ENUNCIADO

En esta práctica actuarás como un ingeniero de datos encargado de analizar y optimizar una aplicación Apache Spark ya desarrollada.

La aplicación procesa un conjunto de datos de ventas utilizando DataFrames de Spark. Tu objetivo no será modificar la lógica de negocio, sino estudiar cómo Spark ejecuta la aplicación, identificar posibles cuellos de botella y evaluar el efecto de distintas configuraciones de ejecución.

Para ello utilizarás Spark UI, los Event Logs y varios parámetros de configuración estudiados en este tema.

La solución completa del ejercicio puede consultarse en el cuaderno M2T4_solucion.ipynb.

ESCENARIO DE TRABAJO

La aplicación trabaja sobre el fichero sales.csv, utilizado anteriormente durante el Tema 2.

El conjunto de datos contiene información de ventas de distintos productos y tiendas, incluyendo ventas, ingresos, stock y promociones.

La aplicación realiza varias agregaciones para obtener indicadores comerciales y guarda el resultado en formato Parquet.

Aunque la aplicación funciona correctamente, se desea analizar si su rendimiento puede mejorarse modificando algunos parámetros de configuración de Spark.

APLICACIÓN SPARK

La aplicación proporcionada realiza automáticamente las siguientes operaciones:

  1. Leer el fichero sales.csv.
  2. Validar y convertir los tipos de datos.
  3. Eliminar registros inconsistentes.
  4. Calcular distintos indicadores de ventas.
  5. Realizar agregaciones por tienda y producto.
  6. Guardar el resultado en formato Parquet.

No es necesario modificar el código de la aplicación durante la práctica. El objetivo será estudiar cómo Spark ejecuta este trabajo. 

spark-submit \
--master local[4] \
--conf spark.app.name=M2T4_Monitorizacion \
app_sales.py \
--input sales.csv \
--output output/base

EJECUCIÓN Y MONITORIZACIÓN

Ejecuta inicialmente la aplicación utilizando la configuración por defecto. Durante la ejecución abre Spark UI desde: 

http://localhost:4040

Observa la información generada por Spark e identifica:

  • Número de Jobs.
  • Número de Stages.
  • Número de Tasks.
  • Duración de cada Stage.
  • Operaciones de Shuffle.
  • Uso de memoria.

Anota estas métricas, ya que servirán como referencia para los experimentos posteriores.

EXPERIMENTOS DE CONFIGURACIÓN

Una vez obtenidas las métricas iniciales, repite la ejecución modificando únicamente un parámetro en cada experimento.

Experimento 1. Paralelismo

--master local[2]

--master local[4]

Compara el número de tareas generadas y el tiempo total de ejecución.

Experimento 2. Número de particiones

--conf spark.sql.shuffle.partitions=8

--conf spark.sql.shuffle.partitions=32

--conf spark.sql.shuffle.partitions=200

Observa cómo cambia el número de Tasks y la duración de los Stages.

Experimento 3. Uso de caché

Activa la caché del DataFrame principal mediante cache() y compara el tiempo total de ejecución.

Experimento 4. Adaptive Query Execution

--conf spark.sql.adaptive.enabled=true

Comprueba si Spark modifica automáticamente el número de particiones tras las operaciones de Shuffle.
