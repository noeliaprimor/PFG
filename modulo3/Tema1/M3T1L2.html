Arquitecturas Lambda y Kappa

  • Descripción
  • Contenido
  • Conclusión

DESCRIPCIÓN

En los sistemas Big Data, uno de los problemas principales consiste en combinar el análisis histórico (batch) con la necesidad de generar resultados con baja latencia (streaming). Para abordar este reto surgieron patrones arquitectónicos orientados al procesamiento de datos a gran escala, entre los que destacan la Arquitectura Lambda y la Arquitectura Kappa.

Apache Spark ofrece un conjunto de librerías y un modelo de ejecución distribuida que permiten implementar ambos enfoques. En particular, Spark SQL facilita el tratamiento de datos estructurados (batch), y Structured Streaming permite el procesamiento continuo de eventos en tiempo (casi) real manteniendo una API unificada basada en DataFrames/Datasets.

CONTENIDO

Arquitectura Lambda

Definición y objetivo

La Arquitectura Lambda persigue combinar lo mejor de dos mundos:

  • Precisión y consistencia mediante procesamiento batch sobre el histórico completo.
  • Baja latencia mediante procesamiento streaming sobre los datos más recientes.

Para ello, separa el procesamiento en capas diferenciadas que operan sobre los mismos datos, pero con diferentes requisitos de latencia y estrategias de cálculo.

Capas principales

Batch Layer

La capa batch procesa el dataset completo (histórico) para generar resultados exactos. Su función principal es construir vistas o agregaciones de alta calidad, asumiendo que el tiempo de cómputo puede ser elevado.

En Spark, esta capa suele implementarse mediante:

  • Jobs batch con Spark SQL / DataFrames.
  • Lectura desde almacenamiento distribuido (HDFS, S3, data lake).
  • Reprocesamiento completo cuando se corrigen errores lógicos o cambian los requisitos de negocio.
Speed Layer

La capa speed (o capa de velocidad) procesa eventos recientes para proporcionar resultados con baja latencia, antes de que el batch se ejecute y “consolide” el estado final. Normalmente opera sobre ventanas temporales cortas y produce resultados aproximados o parciales.

En Spark, esta capa se implementa típicamente con:

  • Structured Streaming (micro-batch o continuous processing, según configuración y versión).
  • Lectura desde sistemas de mensajería (por ejemplo, Kafka) o desde ficheros en modo streaming.
  • Agregaciones incrementales (por ventanas) y actualizaciones de estado.
Serving Layer

La capa serving expone las vistas calculadas para consulta por aplicaciones o usuarios. En Lambda, el serving suele combinar los resultados provenientes de la capa batch y de la capa speed para dar una visión “casi” completa y con baja latencia.

En Spark, la integración con la serving layer suele hacerse escribiendo resultados a sistemas externos:

  • Almacenes analíticos (por ejemplo, bases de datos columnar, data warehouses).
  • Motores de búsqueda/consulta (por ejemplo, Elasticsearch).
  • Bases NoSQL orientadas a lecturas rápidas (por ejemplo, Cassandra).
  • Tablas materializadas o formatos de tablas en data lake (según el diseño del sistema).

Flujo de datos típico

  1. Los eventos se ingieren desde fuentes (aplicaciones, logs, sensores) hacia un bus o cola (p. ej., Kafka).
  2. Los datos se almacenan de forma duradera en un almacenamiento distribuido (histórico).
  3. La capa speed procesa el stream y publica resultados recientes.
  4. La capa batch recalcula periódicamente vistas exactas sobre todo el histórico.
  5. La capa serving ofrece resultados combinados (batch + speed) a consumidores finales.

Ventajas

  • Alta precisión gracias a la recomputación batch completa.
  • Tolerancia a fallos: el histórico permite reprocesar ante errores.
  • Capacidad de corregir y recalcular con cambios en lógica o datos.

Desventajas

  • Duplicación de lógica (batch y streaming pueden requerir transformaciones similares).
  • Mayor complejidad operativa (dos pipelines, coordinación y reconciliación).
  • Coste de mantenimiento más alto (dos modos de ejecución, pruebas duplicadas).

Arquitectura Kappa

Definición y objetivo

La Arquitectura Kappa simplifica Lambda eliminando la separación entre batch y speed. El principio base es tratar todos los datos como un flujo inmutable (log) y construir el procesamiento como una única canalización streaming. Si se necesita recalcular el histórico, se “reproduce” el stream desde el origen (por ejemplo, releyendo Kafka desde offsets anteriores o reinyectando eventos).

Componentes principales

  • Log inmutable de eventos (p. ej., Kafka) que actúa como fuente de verdad.
  • Motor de procesamiento streaming (Structured Streaming) con lógica única.
  • Almacenamiento de resultados para consultas (serving), similar a Lambda, pero alimentado por una sola vía.

Flujo de datos típico

  1. Los eventos se publican en un sistema de log (Kafka) de manera ordenada y durable.
  2. Spark Structured Streaming consume el stream y aplica transformaciones/ventanas/estado.
  3. Los resultados se escriben en un sistema de serving (almacenamiento analítico, base NoSQL, etc.).
  4. Para reprocesar histórico se reejecuta el job consumiendo el log desde el inicio (o desde un offset objetivo).

Ventajas

  • Arquitectura más simple: una única canalización de procesamiento.
  • Menor duplicidad de código y pruebas.
  • Mantenimiento más sencillo y evolución más rápida.
  • Encaja bien con APIs modernas unificadas (DataFrames/Structured Streaming).

Desventajas

  • El reprocesamiento puede ser costoso si el log es muy grande o la retención es limitada.
  • Alta dependencia del sistema de mensajería/log (retención, orden, escalabilidad, consistencia).
  • Para casos extremadamente batch (cargas masivas históricas) puede no ser la opción óptima.

Implementación en Spark: relación con Structured Streaming

De DStreams a Structured Streaming

Spark evolucionó desde enfoques de streaming más antiguos (como DStreams) hacia Structured Streaming, que adopta un modelo declarativo basado en DataFrames y planes de ejecución optimizados. Esta evolución facilita aproximaciones cercanas a Kappa al reducir la brecha entre batch y streaming mediante una API común.

Micro-batch y semántica de procesamiento

Structured Streaming, en su configuración habitual, ejecuta micro-lotes (micro-batches) que procesan intervalos cortos de datos. Esto permite combinar rendimiento y tolerancia a fallos, manteniendo un modelo de “streaming” sobre un motor distribuido.

En diseños Kappa, el mismo pipeline se usa para:

  • Datos “en caliente” (eventos nuevos).
  • Datos históricos (reprocesamiento) releyendo el log o reinyectando eventos.

Comparativa Lambda vs Kappa en Spark

Característica Lambda Kappa
Procesamiento batch Sí (capa batch dedicada) No (todo como streaming)
Procesamiento en tiempo real Sí (capa speed) Sí (pipeline único)
Duplicación de lógica Frecuente Minimizada
Complejidad operativa Alta Media
Reprocesamiento histórico Recomputación batch completa Reproducción del stream
Adecuación con Spark moderno Válida, pero más pesada Muy alineada (API unificada)

Ejemplo conceptual: enfoque Kappa con Spark Structured Streaming

El siguiente fragmento muestra un esquema conceptual de consumo de eventos desde Kafka y escritura de resultados. En una arquitectura Kappa, este pipeline representa la vía principal (y única) de procesamiento. 

from pyspark.sql import SparkSession

spark = (SparkSession.builder
         .appName("Arquitectura-Kappa")
         .getOrCreate())

df_stream = (spark.readStream
             .format("kafka")
             .option("kafka.bootstrap.servers", "localhost:9092")
             .option("subscribe", "eventos")
             .load())

df_transformed = df_stream.selectExpr("CAST(value AS STRING)")

query = (df_transformed.writeStream
         .format("console")
         .start())

query.awaitTermination()
CONCLUSIÓN

Resumen: Lambda y Kappa son patrones útiles con trade-offs claros; la elección debe justificarse según requisitos de latencia, precisión y coste.
