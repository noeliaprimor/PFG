Adquisición y transmisión de eventos

  • Introducción
  • Contenido
  • Conclusión

INTRODUCCIÓN

En arquitecturas modernas de datos, la adquisición y transmisión de eventos constituyen la primera fase del procesamiento en tiempo real. Antes de que un motor como Spark Structured Streaming pueda analizar la información, los eventos deben ser capturados desde su origen y transportados de forma fiable hacia el sistema de procesamiento.

Esta fase es crítica, ya que impacta directamente en la latencia, consistencia, escalabilidad y resiliencia de la solución completa.

El objetivo de esta lección es entender cómo se producen, capturan, transportan y preparan los eventos antes de llegar a una herramienta de procesamiento como Spark. Una buena estrategia de adquisición evita pérdidas de datos, reduce duplicados y permite escalar el sistema cuando aumenta el volumen de información.

CONTENIDO

¿Qué es un evento?

Un evento es una unidad de información que describe algo que ha ocurrido en un sistema. Generalmente incluye:

  • Identificador o clave.
  • Payload o contenido (datos del evento).
  • Timestamp (momento en que ocurrió).
  • Metadatos adicionales.

En la práctica, un evento debe representar un hecho ya ocurrido, no una orden ni una intención futura. Por ejemplo, pedido_creado, pago_confirmado o sensor_temperatura_leido son buenos nombres porque describen hechos concretos del dominio.

Anatomía habitual de un evento 
{
  "event_id": "evt-2026-0001",
  "event_type": "pedido_creado",
  "event_time": "2026-05-08T10:15:30Z",
  "source": "checkout-service",
  "key": "pedido-8421",
  "payload": {
    "cliente_id": "cli-91",
    "importe": 129.95,
    "moneda": "EUR"
  },
  "metadata": {
    "version": "1.0",
    "trace_id": "trc-556"
  }
}

El campo event_id ayuda a detectar duplicados, event_type permite enrutar o clasificar eventos, event_time indica cuándo ocurrió realmente el hecho y payload contiene los datos de negocio.

Ejemplos de eventos:

  • Una transacción financiera.
  • Un clic en una aplicación web.
  • Una lectura de sensor IoT.
  • Un log generado por un servidor.

Tiempo de evento y tiempo de procesamiento

En streaming conviene distinguir entre dos momentos diferentes:

  • Tiempo de evento: instante en el que el hecho ocurrió en el sistema origen.
  • Tiempo de procesamiento: instante en el que Spark recibe y procesa el evento.

Esta diferencia es importante porque los eventos pueden llegar tarde, repetidos o fuera de orden. Por ejemplo, un sensor puede quedarse sin conexión durante unos minutos y enviar sus lecturas acumuladas más tarde. Aunque Spark las procese después, su análisis debería respetar el momento real en el que se generaron.

Fuentes de eventos

  • Aplicaciones web y móviles.
  • Microservicios.
  • Sistemas financieros.
  • Sensores IoT.
  • Sistemas legacy.

Las fuentes pueden generar eventos de forma continua y a gran velocidad, lo que requiere sistemas intermedios capaces de absorber picos de carga.

Formas de adquisición de eventos

Ingesta directa desde aplicaciones

La propia aplicación publica eventos cada vez que ocurre una acción relevante. Es un patrón habitual en microservicios, comercio electrónico, banca digital o plataformas SaaS. Permite capturar eventos con baja latencia, aunque exige cuidar la fiabilidad del productor.

Captura desde logs

Algunos sistemas generan logs que posteriormente son recogidos por agentes como Filebeat, Fluent Bit o Logstash. Esta aproximación es útil para observabilidad, auditoría y análisis de comportamiento, pero requiere normalizar el formato de los registros.

CDC (Change Data Capture)

La captura de cambios en bases de datos convierte inserciones, actualizaciones y borrados en eventos. Herramientas como Debezium permiten publicar esos cambios en Kafka para que otros sistemas los consuman sin consultar continuamente la base de datos.

Ingesta IoT

En escenarios IoT, los dispositivos suelen enviar datos mediante protocolos ligeros como MQTT o HTTP. Un gateway puede validar, enriquecer y reenviar las lecturas hacia Kafka u otro sistema de mensajería antes de que sean procesadas por Spark.

Diseño del evento

Diseñar correctamente el evento es tan importante como elegir la tecnología de transporte. Un evento mal definido puede complicar el procesamiento posterior, provocar ambigüedad o hacer difícil evolucionar el sistema.

Elemento Función Recomendación
Clave Determina agrupación, orden y particionado. Usar una clave estable, como id de pedido, cliente o dispositivo.
Timestamp Permite ordenar y calcular ventanas temporales. Guardar el tiempo de evento en UTC y con formato consistente.
Payload Contiene la información de negocio. Evitar campos innecesarios y documentar unidades, tipos y significado.
Versión Facilita la evolución del esquema. Versionar cambios incompatibles y mantener compatibilidad hacia atrás.

También es recomendable definir un esquema de datos mediante JSON Schema, Avro o Protobuf. Esto reduce errores entre productores y consumidores, especialmente cuando varios equipos publican y consumen eventos del mismo dominio.

Sistemas de transmisión de eventos

Sistemas de mensajería

Plataformas como Apache Kafka permiten desacoplar productores y consumidores, garantizando almacenamiento persistente y distribución escalable.

Brokers de mensajes

Actúan como intermediarios que reciben eventos de productores y los distribuyen a consumidores interesados.

APIs REST o Webhooks

Algunos sistemas transmiten eventos mediante llamadas HTTP, aunque suelen complementarse con sistemas de mensajería para mayor escalabilidad.

Colas tradicionales

Soluciones como RabbitMQ o ActiveMQ se utilizan cuando interesa una entrega orientada a cola, con confirmaciones explícitas y patrones de enrutamiento flexibles. Son útiles para tareas asíncronas y comunicación entre servicios, aunque no siempre ofrecen el mismo modelo de retención y reprocesamiento que Kafka.

Gateways y agentes de ingesta

En muchos sistemas existe una capa intermedia que valida credenciales, normaliza formatos, añade metadatos y aplica reglas básicas antes de publicar los eventos. Esta capa evita que Spark reciba datos incompletos o con formatos heterogéneos.

Flujo típico de adquisición y transmisión

En una arquitectura real, este flujo suele incorporar validación, serialización, compresión, autenticación, monitorización y mecanismos de reintento. Estas piezas no modifican el objetivo principal, pero sí determinan si el pipeline será fiable cuando aparezcan fallos o picos de carga.

Productores y consumidores

Productores

Son los componentes que generan y publican eventos. Un productor debe encargarse de serializar los datos, elegir una clave adecuada, gestionar reintentos y confirmar que el evento ha sido aceptado por el sistema de mensajería.

Consumidores

Son los procesos que leen eventos para transformarlos, almacenarlos o reaccionar ante ellos. Spark Structured Streaming actúa como consumidor cuando lee desde Kafka, procesa los registros de forma continua y mantiene el progreso mediante offsets y checkpoints.

Desacoplamiento

Una ventaja clave de usar un sistema de transmisión es que productores y consumidores no tienen que ejecutarse al mismo ritmo. Kafka puede almacenar temporalmente los eventos mientras Spark los procesa, lo que permite absorber picos de carga y recuperar el consumo tras una parada.

Requisitos clave en la transmisión de eventos

Escalabilidad

El sistema debe soportar incrementos en el volumen de eventos sin degradar el rendimiento. Para conseguirlo se distribuyen los datos en particiones y se añaden consumidores en paralelo.

Tolerancia a fallos

Los eventos no deben perderse ante fallos de red o caída de nodos. Esto requiere replicación, confirmaciones de escritura y reintentos controlados.

Orden

En muchos casos es importante mantener el orden dentro de una clave específica. Kafka garantiza orden dentro de una partición, por lo que la elección de la clave influye directamente en este requisito.

Persistencia

Los eventos deben almacenarse durante un tiempo determinado para permitir reprocesamiento.

Garantías de entrega
  • At-most-once: el evento puede perderse, pero no se procesa más de una vez.
  • At-least-once: el evento no debería perderse, aunque puede procesarse más de una vez.
  • Exactly-once: busca evitar pérdidas y duplicados, pero depende de la configuración y del sink utilizado.
Latencia

La latencia mide el tiempo transcurrido desde que el evento ocurre hasta que el resultado está disponible. Depende del productor, la red, el broker, el motor de procesamiento y el destino final.

Backpressure

Se produce cuando los eventos llegan más rápido de lo que pueden procesarse. En estos casos, el sistema debe regular la velocidad de consumo, escalar recursos o almacenar temporalmente los datos sin perderlos.

Serialización y formatos

Antes de transmitirse, los eventos se convierten a un formato concreto. La elección afecta al rendimiento, compatibilidad y facilidad de depuración.

  • JSON: legible y sencillo, aunque menos eficiente en tamaño y validación.
  • Avro: compacto, orientado a esquemas y muy usado con Kafka.
  • Protobuf: eficiente y adecuado para contratos estrictos entre servicios.
  • CSV: simple, pero menos recomendable para eventos complejos o anidados.

En entornos productivos es común combinar Avro o Protobuf con un registro de esquemas para controlar la evolución de los eventos y evitar incompatibilidades entre versiones.

Integración con Spark Structured Streaming

Spark puede consumir eventos desde sistemas como Kafka utilizando la API readStream. 

stream_df = (spark
    .readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "localhost:9092")
    .option("subscribe", "mi_topic")
    .load())

El DataFrame resultante contiene columnas propias de Kafka, como key, value, topic, partition, offset y timestamp. Normalmente, el contenido del evento llega en la columna value en formato binario, por lo que debe convertirse y parsearse. 

from pyspark.sql.functions import col, from_json
from pyspark.sql.types import StructType, StructField, StringType, DoubleType

schema = StructType([
    StructField("event_id", StringType()),
    StructField("event_type", StringType()),
    StructField("event_time", StringType()),
    StructField("key", StringType()),
    StructField("payload", StructType([
        StructField("cliente_id", StringType()),
        StructField("importe", DoubleType()),
        StructField("moneda", StringType())
    ]))
])

eventos_df = (stream_df
    .selectExpr("CAST(key AS STRING)", "CAST(value AS STRING) AS json_value", "timestamp")
    .select(from_json(col("json_value"), schema).alias("evento"), "timestamp")
    .select("evento.*", "timestamp"))

Una vez adquiridos y parseados, los eventos pueden filtrarse, enriquecerse, agregarse por ventanas temporales o almacenarse mediante writeStream.

Control de offsets y checkpoints

Spark utiliza offsets para saber qué mensajes de Kafka ya han sido leídos. Para recuperarse correctamente ante fallos, Structured Streaming guarda esta información en una ubicación de checkpoint. Sin checkpoint, una aplicación puede repetir trabajo o perder la posición de consumo. 

query = (eventos_df
    .writeStream
    .format("parquet")
    .option("path", "/datos/eventos")
    .option("checkpointLocation", "/checkpoints/eventos")
    .start())

La ubicación de checkpoint debe estar en un almacenamiento persistente y accesible por el clúster, no en un directorio temporal local que pueda desaparecer al reiniciar el proceso.

Problemas comunes en adquisición de eventos

  • Eventos duplicados.
  • Pérdida de eventos por mala configuración.
  • Desorden temporal (late events).
  • Cuellos de botella en productores.
  • Esquemas incompatibles entre productores y consumidores.
  • Particiones desbalanceadas por una mala elección de clave.
  • Retención insuficiente para reprocesar datos tras una incidencia.

Buenas prácticas

  • Diseñar correctamente particiones en el sistema de mensajería.
  • Definir claves adecuadas para balancear carga.
  • Configurar retención suficiente para reprocesamiento.
  • Implementar monitorización continua.
  • Evitar dependencias síncronas innecesarias.
  • Usar identificadores únicos para facilitar deduplicación.
  • Separar eventos por dominio o tipo en topics bien definidos.
  • Validar los eventos antes de publicarlos.
  • Documentar contratos de datos y versionar esquemas.
  • Configurar checkpoints persistentes en las aplicaciones de Spark.

Ejemplo de arquitectura completa

  1. Una aplicación de comercio electrónico genera eventos de compra.
  2. El productor publica cada evento en un topic Kafka llamado pedidos.
  3. Kafka distribuye los eventos por particiones usando el identificador de pedido como clave.
  4. Spark Structured Streaming consume el topic, parsea el JSON y valida el esquema.
  5. Los eventos se enriquecen con información de clientes o productos.
  6. Los resultados se escriben en un Data Lake y en una tabla analítica.
  7. Un dashboard consulta los datos procesados para mostrar métricas casi en tiempo real.

Este tipo de arquitectura permite separar responsabilidades: la aplicación solo publica eventos, Kafka se encarga del transporte y retención, Spark procesa los datos y los sistemas de destino almacenan resultados listos para análisis.

CONCLUSIÓN

La adquisición y transmisión de eventos es la base de cualquier arquitectura de procesamiento en tiempo real. Una correcta elección de tecnologías, configuración adecuada y diseño escalable garantizan que el sistema pueda absorber grandes volúmenes de datos sin comprometer la fiabilidad.

En combinación con motores como Spark, estos sistemas permiten construir pipelines robustos, resilientes y orientados a análisis en tiempo real.
