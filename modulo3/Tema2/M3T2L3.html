Sinks y conectores

  • Introducción
  • Contenido
  • Conclusión

INTRODUCCIÓN

En una arquitectura de datos, los conectores permiten integrar Spark con sistemas externos, mientras que los sinks representan los destinos donde se almacenan o envían los resultados del procesamiento.

En el contexto de procesamiento batch y streaming, los conectores y sinks son fundamentales para construir pipelines completos que integren fuentes de datos, motores de procesamiento y sistemas de almacenamiento o visualización.

Elegir correctamente un conector o un sink no es solo una decisión técnica de compatibilidad. También afecta a la consistencia, latencia, coste, seguridad, recuperación ante fallos y facilidad de explotación del pipeline.

CONTENIDO

Conceptos fundamentales

Conector

Componente que permite a Spark leer o escribir datos desde o hacia sistemas externos (bases de datos, sistemas de mensajería, almacenamiento cloud, etc.).

Un conector traduce la API de Spark al protocolo del sistema externo. Por ejemplo, el conector JDBC convierte operaciones de Spark en consultas SQL, mientras que el conector Kafka convierte lecturas y escrituras en operaciones sobre topics, particiones y offsets.

Sink

Destino final donde se escriben los datos procesados. En streaming, el sink define cómo se materializan los resultados de una consulta continua.

El sink puede ser definitivo, como un Data Lake o una base de datos analítica, o intermedio, como otro topic Kafka que alimentará un segundo pipeline.

Source

Aunque esta lección se centra en sinks y conectores, también conviene recordar el concepto de source: el origen desde el que Spark lee datos. En Structured Streaming, una consulta suele tener una source, un conjunto de transformaciones y un sink.

Criterios para elegir conectores y sinks

  • Semántica: si permite escrituras idempotentes, transacciones o solo append.
  • Latencia: si el destino necesita resultados en segundos, minutos o modo batch.
  • Formato: si el sistema trabaja mejor con Parquet, Delta, JSON, Avro o tablas SQL.
  • Escalabilidad: si soporta escritura concurrente desde varios ejecutores.
  • Recuperación: si funciona correctamente con checkpoints y reintentos.
  • Seguridad: si permite autenticación, cifrado y control de permisos.

Conectores en Spark (Batch)

Archivos y formatos soportados
  • Parquet
  • ORC
  • CSV
  • JSON
  • Avro (mediante dependencia adicional)

Los formatos columnares como Parquet y ORC suelen ser preferibles para analítica porque reducen tamaño en disco y permiten leer solo las columnas necesarias. JSON y CSV son cómodos para intercambio o depuración, pero resultan menos eficientes en grandes volúmenes.

JDBC

Permite conectarse a bases de datos relacionales. 

df = (spark.read
    .format("jdbc")
    .option("url", "jdbc:postgresql://host:5432/db")
    .option("dbtable", "tabla")
    .option("user", "usuario")
    .option("password", "password")
    .load())

En lecturas grandes por JDBC es recomendable particionar la lectura para evitar que una sola conexión concentre todo el trabajo. 

df = (spark.read
    .format("jdbc")
    .option("url", "jdbc:postgresql://host:5432/db")
    .option("dbtable", "ventas")
    .option("user", "usuario")
    .option("password", "password")
    .option("partitionColumn", "id")
    .option("lowerBound", "1")
    .option("upperBound", "1000000")
    .option("numPartitions", "8")
    .load())

Esta configuración permite que Spark divida la lectura en rangos y ejecute varias tareas en paralelo, reduciendo el tiempo total de ingesta.

Almacenamiento en la nube
  • AWS S3
  • Azure Data Lake
  • Google Cloud Storage

En almacenamiento cloud es importante configurar credenciales, permisos, endpoint correcto, políticas de ciclo de vida y formato de salida. También hay que vigilar el problema de muchos archivos pequeños, especialmente en streaming.

Escritura batch con Spark

En modo batch, Spark escribe un DataFrame completo y finaliza el trabajo. Es común utilizar mode para controlar qué ocurre si el destino ya existe. 

(df.write
    .format("parquet")
    .mode("overwrite")
    .partitionBy("fecha")
    .save("/data/ventas_parquet"))
  • append: añade datos al destino.
  • overwrite: sustituye datos existentes.
  • ignore: no escribe si el destino existe.
  • error: falla si el destino ya existe.

Conectores en Structured Streaming

Fuentes comunes
  • Kafka
  • Socket
  • Archivos (monitorización de directorios)
Ejemplo lectura desde Kafka 
stream_df = (spark
    .readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "localhost:9092")
    .option("subscribe", "mi_topic")
    .load())

En streaming, Spark no carga un conjunto finito de datos, sino que mantiene una consulta activa. Cada microbatch procesa los registros disponibles desde la última posición confirmada.

Lectura de archivos en streaming

Spark también puede monitorizar un directorio y procesar nuevos archivos conforme aparecen. Es útil para integraciones con sistemas que depositan ficheros periódicamente. 

stream_files_df = (spark
    .readStream
    .schema(schema)
    .format("json")
    .load("/landing/eventos"))

En este patrón se debe definir el esquema de forma explícita. Inferir esquemas automáticamente en streaming puede ser costoso y provocar comportamientos inconsistentes.

Sinks en Structured Streaming

Console Sink

Utilizado principalmente para desarrollo y pruebas. 

query = (df
    .writeStream
    .format("console")
    .start())

No debe usarse como destino productivo porque imprime resultados en la salida estándar y no ofrece persistencia ni garantías útiles para recuperación.

File Sink (Parquet, CSV, etc.) 
query = (df
    .writeStream
    .format("parquet")
    .option("path", "/output")
    .option("checkpointLocation", "/checkpoint")
    .start())

El file sink es habitual para construir zonas de datos en un Data Lake. Parquet suele ser una opción adecuada para analítica por su compresión y lectura columnar.

Kafka Sink

Permite publicar resultados nuevamente en un topic Kafka. 

query = (df
    .writeStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "localhost:9092")
    .option("topic", "output_topic")
    .option("checkpointLocation", "/checkpoint")
    .start())

Para escribir en Kafka, el DataFrame debe contener una columna value y, opcionalmente, key, ambas en formato binario o string. Si se quiere publicar un evento estructurado, normalmente se convierte a JSON antes de enviarlo. 

from pyspark.sql.functions import col, to_json, struct

kafka_output_df = df.select(
    col("pedido_id").cast("string").alias("key"),
    to_json(struct("pedido_id", "cliente_id", "importe", "fecha")).alias("value")
)

query = (kafka_output_df
    .writeStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "localhost:9092")
    .option("topic", "pedidos_procesados")
    .option("checkpointLocation", "/checkpoint/pedidos_procesados")
    .start())
Foreach Sink

Permite definir lógica personalizada para escribir resultados (por ejemplo, hacia una API externa o sistema especializado).

ForeachBatch

foreachBatch permite ejecutar lógica batch sobre cada microbatch. Es útil para escribir en sistemas que no tienen sink nativo de streaming o para aplicar operaciones transaccionales por lote. 

def escribir_microbatch(batch_df, batch_id):
    (batch_df.write
        .format("jdbc")
        .option("url", "jdbc:postgresql://host:5432/db")
        .option("dbtable", "eventos_procesados")
        .option("user", "usuario")
        .option("password", "password")
        .mode("append")
        .save())

query = (df.writeStream
    .foreachBatch(escribir_microbatch)
    .option("checkpointLocation", "/checkpoint/jdbc")
    .start())

Al usar foreachBatch, la función debe diseñarse para tolerar reintentos. Si un microbatch se ejecuta de nuevo tras un fallo, el destino no debería quedar con duplicados incoherentes.

Checkpointing

En streaming, los sinks requieren una ubicación de checkpoint para almacenar offsets y estado, permitiendo recuperación ante fallos. 

.option("checkpointLocation", "/ruta/checkpoint")

El checkpoint debe estar en almacenamiento persistente y exclusivo para cada consulta. Reutilizar el mismo directorio para consultas diferentes puede corromper el estado o provocar resultados incorrectos.

También es importante no borrar checkpoints en producción sin una estrategia clara. Si se elimina el checkpoint, Spark puede perder la referencia de offsets procesados y empezar desde una posición distinta según la configuración de la fuente.

Modos de salida

  • Append
  • Update
  • Complete

El modo elegido afecta cómo se escriben los datos en el sink.

Append

Solo escribe filas nuevas cuando Spark considera que ya no cambiarán. Es habitual en flujos sin agregaciones o con ventanas cerradas mediante watermark.

Update

Escribe únicamente las filas que han cambiado desde el último microbatch. Puede ser útil en agregaciones incrementales.

Complete

Escribe todo el resultado agregado en cada microbatch. Puede ser costoso si el estado crece mucho.

Triggers

Los triggers controlan cada cuánto se ejecuta una consulta de streaming. Elegirlos bien ayuda a equilibrar latencia y coste de procesamiento. 

query = (df.writeStream
    .format("parquet")
    .option("path", "/output")
    .option("checkpointLocation", "/checkpoint")
    .trigger(processingTime="30 seconds")
    .start())

Un trigger muy frecuente reduce latencia, pero puede aumentar la sobrecarga. Un trigger más espaciado reduce coste, aunque los datos tardarán más en estar disponibles.

Consideraciones de rendimiento

  • Evitar demasiados archivos pequeños.
  • Configurar particiones adecuadas.
  • Optimizar número de particiones en Kafka.
  • Usar formatos columnares (Parquet) para analítica.
  • Ajustar el número de particiones antes de escribir en destinos sensibles.
  • Evitar escrituras fila a fila hacia bases de datos externas.
  • Controlar la frecuencia de microbatches mediante triggers.

En sinks de archivos, el exceso de archivos pequeños complica consultas posteriores y aumenta costes de metadatos. En sinks JDBC, demasiadas conexiones simultáneas pueden saturar la base de datos. En Kafka, un mal particionado puede concentrar carga en pocas particiones.

Seguridad en conectores

  • Uso de SSL/TLS en Kafka.
  • Credenciales seguras en JDBC.
  • Gestión de secretos mediante variables de entorno o vaults.

Las credenciales no deberían quedar escritas directamente en notebooks o ficheros HTML de ejemplo usados en producción. En entornos reales conviene usar gestores de secretos, variables de entorno o mecanismos propios de la plataforma cloud.

Consistencia e idempotencia

Un sink es idempotente cuando repetir la misma escritura no cambia el resultado final de forma incorrecta. Esta propiedad es muy importante en streaming porque un microbatch puede repetirse tras un fallo.

  • En Kafka, usar claves estables ayuda a compactar o deduplicar resultados posteriores.
  • En bases de datos, pueden usarse claves primarias, UPSERT o tablas intermedias.
  • En Data Lakes, formatos transaccionales como Delta Lake facilitan operaciones consistentes.

Si el sink no es idempotente, el pipeline debe incorporar deduplicación mediante identificadores de evento, control de lotes o lógica transaccional en el destino.

Arquitectura típica con conectores

  1. Fuente (Kafka).
  2. Procesamiento (Spark).
  3. Sink (Data Lake, Base de datos, Kafka, Dashboard).

Una arquitectura más completa puede incluir una zona raw donde se guardan los eventos originales, una zona procesada con datos limpios y una zona de consumo optimizada para dashboards, modelos analíticos o APIs. 

Kafka topic
    -> Spark Structured Streaming
    -> Data Lake raw
    -> Transformaciones y validaciones
    -> Data Lake curado / Base analítica
    -> Dashboard o aplicación de consumo

Errores comunes

  • Usar console sink como si fuera un destino real.
  • No configurar checkpointLocation en consultas de streaming.
  • Escribir demasiados archivos pequeños en almacenamiento cloud.
  • Saturar una base de datos relacional con escrituras paralelas sin control.
  • Guardar credenciales en texto plano dentro del código.
  • No diseñar deduplicación cuando se trabaja con semántica at-least-once.

Buenas prácticas

  • Separar entorno de desarrollo y producción.
  • Monitorizar latencia y throughput.
  • Configurar correctamente checkpointing.
  • Validar compatibilidad de versiones.
  • Preferir formatos columnares para analítica.
  • Diseñar escrituras idempotentes siempre que sea posible.
  • Documentar contratos de datos entre fuentes, Spark y destinos.
  • Probar recuperación ante fallos antes de pasar a producción.
  • Dimensionar el sink, no solo el clúster Spark.

CONCLUSIÓN

Los conectores y sinks permiten integrar Spark dentro de arquitecturas de datos complejas, facilitando la comunicación entre sistemas.

Una correcta selección y configuración de conectores y sinks es esencial para garantizar rendimiento, fiabilidad y escalabilidad en entornos Big Data.
