Casos prácticos con Spark Streaming

  • Introducción
  • Contenido
  • Conclusión

INTRODUCCIÓN

En esta lección se presentan casos prácticos implementables con Apache Spark Structured Streaming utilizando Apache Kafka como fuente de eventos y Parquet como almacenamiento de salida (data lake). Este enfoque es habitual en arquitecturas modernas (incluyendo patrones tipo Kappa) debido a que Kafka actúa como un log distribuido y Parquet ofrece un formato columnar eficiente para analítica posterior.

Los ejemplos están diseñados para un entorno típico de laboratorio: un clúster Spark (por ejemplo, Standalone en Docker Compose), un broker Kafka accesible desde Spark, y un sistema de ficheros (local o volumen montado) donde se persisten los resultados y el estado del streaming (checkpoints).

El objetivo no es solo ejecutar código, sino aprender a razonar sobre decisiones de arquitectura: qué datos conservar en bruto, qué métricas calcular en tiempo real, cómo controlar eventos tardíos, cómo diseñar checkpoints y cómo verificar que los resultados son reproducibles.

CONTENIDO

Consideraciones generales del entorno

  • Fuente: Kafka (topics con eventos en JSON).
  • Salida: Parquet en un directorio de data lake (por ejemplo, /data/out/...).
  • Estado y tolerancia a fallos: checkpoints en /data/chk/....
  • Tiempo de evento: se asume un campo ts de tipo timestamp (event time) dentro del JSON.

Criterios para seleccionar un caso práctico

Antes de implementar un pipeline conviene definir qué se quiere medir y qué nivel de latencia es aceptable. No todos los casos necesitan el mismo diseño.

  • Clickstream: prioriza volumen, ventanas cortas y métricas agregadas.
  • Fraude: prioriza detección rápida, reglas claras y baja latencia.
  • Bronze-Silver: prioriza calidad, deduplicación, trazabilidad y datos reutilizables.

También es importante decidir si los resultados se necesitan para una alerta inmediata, para analítica posterior o para alimentar otro sistema de eventos.

Base técnica: lectura desde Kafka y parseo de JSON

Structured Streaming lee de Kafka devolviendo un DataFrame con columnas como key, value, topic, partition, offset y timestamp (timestamp de Kafka). Habitualmente, el payload de negocio se encuentra en value y se parsea desde JSON a columnas estructuradas con un esquema explícito.

Plantilla de lectura y parseo (PySpark) 
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, from_json
from pyspark.sql.types import StructType, StructField, StringType, TimestampType, DoubleType

spark = SparkSession.builder.appName("TFG-Streaming-Kafka-Parquet").getOrCreate()

bootstrap_servers = "kafka:9092"  # ajustar al hostname/puerto accesible desde Spark

raw = (spark.readStream
       .format("kafka")
       .option("kafka.bootstrap.servers", bootstrap_servers)
       .option("subscribe", "mi_topic")
       .option("startingOffsets", "latest")   # "earliest" para reprocesar desde el inicio
       .load())

schema = StructType([
    StructField("ts", TimestampType(), True),
    StructField("id", StringType(), True),
    StructField("value", DoubleType(), True),
    StructField("type", StringType(), True)
])

events = (raw.select(from_json(col("value").cast("string"), schema).alias("e"))
          .select("e.*"))

A partir de este patrón se construyen los tres casos prácticos del capítulo.

Validación común de eventos

Los tres casos deben validar datos antes de agregarlos o escribirlos. Una validación mínima suele incluir:

  • Comprobar que el timestamp de evento existe y se puede parsear.
  • Eliminar o enviar a cuarentena registros sin identificador.
  • Validar rangos numéricos, por ejemplo importes no negativos.
  • Registrar eventos descartados para poder auditar pérdidas de calidad.
valid_events = events.dropna(subset=["ts", "id"])
invalid_events = events.filter(col("ts").isNull() | col("id").isNull())

Caso práctico 1: Clickstream web (usuarios activos y páginas top) a Parquet

1. Objetivo

Analizar eventos de navegación en tiempo real para calcular:

  • Usuarios activos por minuto (aproximación eficiente).
  • Top URLs por ventana de 5 minutos.
2. Topic y esquema de eventos

Topic Kafka sugerido: clicks

Ejemplo de evento (JSON): 

{
  "ts": "2026-03-04T10:05:12Z",
  "userId": "u123",
  "url": "/pricing"
}
3. Implementación (PySpark) 
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, from_json, window, approx_count_distinct
from pyspark.sql.types import StructType, StructField, StringType, TimestampType

spark = SparkSession.builder.appName("Case1-Clickstream").getOrCreate()

bootstrap_servers = "kafka:9092"
topic = "clicks"

schema = StructType([
    StructField("ts", TimestampType(), True),
    StructField("userId", StringType(), True),
    StructField("url", StringType(), True),
])

raw = (spark.readStream
       .format("kafka")
       .option("kafka.bootstrap.servers", bootstrap_servers)
       .option("subscribe", topic)
       .option("startingOffsets", "latest")
       .load())

events = (raw.select(from_json(col("value").cast("string"), schema).alias("e"))
          .select("e.*")
          .dropna(subset=["ts", "userId", "url"]))

# 1) Usuarios activos por minuto (usando watermark para controlar datos tardíos)
active_users_1m = (events
  .withWatermark("ts", "10 minutes")
  .groupBy(window(col("ts"), "1 minute"))
  .agg(approx_count_distinct("userId").alias("active_users")))

# 2) Top URLs por ventana de 5 minutos
urls_5m = (events
  .withWatermark("ts", "10 minutes")
  .groupBy(window(col("ts"), "5 minutes"), col("url"))
  .count())

# Escritura a Parquet (dos sinks independientes)
q1 = (active_users_1m.writeStream
      .outputMode("append")
      .format("parquet")
      .option("path", "/data/out/case1_active_users_1m")
      .option("checkpointLocation", "/data/chk/case1_active_users_1m")
      .start())

q2 = (urls_5m.writeStream
      .outputMode("append")
      .format("parquet")
      .option("path", "/data/out/case1_top_urls_5m")
      .option("checkpointLocation", "/data/chk/case1_top_urls_5m")
      .start())

spark.streams.awaitAnyTermination()
4. Aspectos clave
  • Watermark: limita el estado en memoria y permite “cerrar” ventanas cuando se supera el umbral de tardanza.
  • Salida en modo append: adecuada cuando las ventanas se consideran completas (con watermark).
  • Dos salidas Parquet: facilita consumo analítico separado (métricas de usuarios vs ranking de URLs).
5. Posibles ampliaciones
  • Calcular sesiones de usuario agrupando eventos cercanos en el tiempo.
  • Separar tráfico por dispositivo, país o campaña.
  • Publicar alertas en Kafka cuando una URL supere un umbral de visitas.

Caso práctico 2: Detección de fraude simple (picos de transacciones) y persistencia de alertas

1. Objetivo

Detectar patrones sospechosos basados en volumen de transacciones por tarjeta en ventanas cortas: si una tarjeta supera un umbral de operaciones en un intervalo, se genera una alerta.

2. Topic y esquema de eventos

Topic Kafka sugerido: transactions

Ejemplo de evento (JSON): 

{
  "ts": "2026-03-04T10:07:40Z",
  "txId": "t999",
  "cardId": "c100",
  "amount": 35.50,
  "merchant": "shop_01"
}
3. Implementación (PySpark) 
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, from_json, window, lit
from pyspark.sql.types import StructType, StructField, StringType, TimestampType, DoubleType

spark = SparkSession.builder.appName("Case2-Fraud-Bursts").getOrCreate()

bootstrap_servers = "kafka:9092"
topic = "transactions"

schema = StructType([
    StructField("ts", TimestampType(), True),
    StructField("txId", StringType(), True),
    StructField("cardId", StringType(), True),
    StructField("amount", DoubleType(), True),
    StructField("merchant", StringType(), True),
])

raw = (spark.readStream
       .format("kafka")
       .option("kafka.bootstrap.servers", bootstrap_servers)
       .option("subscribe", topic)
       .option("startingOffsets", "latest")
       .load())

tx = (raw.select(from_json(col("value").cast("string"), schema).alias("t"))
      .select("t.*")
      .dropna(subset=["ts", "cardId", "txId"]))

# Regla: más de N transacciones por cardId en 2 minutos (ventana deslizante)
N = 10
alerts = (tx
  .withWatermark("ts", "15 minutes")
  .groupBy(window(col("ts"), "2 minutes", "30 seconds"), col("cardId"))
  .count()
  .filter(col("count") >= lit(N))
  .withColumn("rule", lit("BURST_TX_COUNT"))
  .withColumn("threshold", lit(N)))

q = (alerts.writeStream
     .outputMode("append")
     .format("parquet")
     .option("path", "/data/out/case2_fraud_alerts")
     .option("checkpointLocation", "/data/chk/case2_fraud_alerts")
     .start())

q.awaitTermination()
4. Aspectos clave
  • Ventana deslizante: permite detectar picos aunque no coincidan exactamente con bordes fijos de ventana.
  • Salida Parquet de alertas: se integra fácilmente con análisis posterior (dashboards, reporting, auditoría).
  • Watermark: evita que el estado crezca indefinidamente en presencia de eventos tardíos.
5. Posibles ampliaciones
  • Añadir reglas por importe total acumulado, no solo por número de transacciones.
  • Incorporar una tabla estática de tarjetas bloqueadas o comercios de riesgo.
  • Escribir alertas en un topic Kafka para que otro servicio las consuma.

Caso práctico 3: ETL continuo Bronze → Silver con foreachBatch

1. Objetivo

Construir un pipeline de streaming que:

  • Consuma eventos crudos desde Kafka (capa Bronze).
  • Limpie, valide y deduplique por micro-lote.
  • Escriba a Parquet particionado (capa Silver) con mayor control del proceso.
2. Topic y esquema de eventos

Topic Kafka sugerido: metrics_raw

Ejemplo de evento (JSON): 

{
  "ts": "2026-03-04T10:09:05Z",
  "id": "m001",
  "source": "appA",
  "value": 12.7
}
3. Implementación (PySpark) 
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, from_json
from pyspark.sql.types import StructType, StructField, StringType, TimestampType, DoubleType

spark = SparkSession.builder.appName("Case3-Bronze-Silver-ETL").getOrCreate()

bootstrap_servers = "kafka:9092"
topic = "metrics_raw"

schema = StructType([
    StructField("ts", TimestampType(), True),
    StructField("id", StringType(), True),
    StructField("source", StringType(), True),
    StructField("value", DoubleType(), True),
])

raw = (spark.readStream
       .format("kafka")
       .option("kafka.bootstrap.servers", bootstrap_servers)
       .option("subscribe", topic)
       .option("startingOffsets", "latest")
       .load())

events = (raw.select(from_json(col("value").cast("string"), schema).alias("e"))
          .select("e.*"))

def process_batch(batch_df, batch_id: int):
    # Validación y limpieza básica
    clean = (batch_df
             .dropna(subset=["ts", "id", "source", "value"])
             .dropDuplicates(["id"])  # ejemplo: deduplicación simple por id
             .withColumn("date", col("ts").cast("date")))

    # Escritura a Silver (particionado por fecha)
    (clean.write
     .mode("append")
     .partitionBy("date")
     .parquet("/data/out/case3_silver_metrics"))

q = (events.writeStream
     .foreachBatch(process_batch)
     .option("checkpointLocation", "/data/chk/case3_bronze_to_silver")
     .start())

q.awaitTermination()
4. Aspectos clave
  • foreachBatch: permite aplicar lógica batch (deduplicación, validación, escritura particionada) dentro de un pipeline streaming.
  • Particionado por fecha: mejora la eficiencia de consultas posteriores y reduce el coste de lectura.
  • Checkpoints: necesarios para tolerancia a fallos y recuperación del progreso del streaming.
5. Posibles ampliaciones
  • Crear una salida de cuarentena para registros inválidos.
  • Guardar métricas de calidad por micro-lote.
  • Usar Delta Lake para aplicar MERGE y deduplicación idempotente.

Buenas prácticas para Kafka + Parquet en Structured Streaming

Checkpointing obligatorio
  • Definir siempre checkpointLocation en sinks persistentes.
  • Mantener un directorio de checkpoint por consulta (q1, q2, etc.) para evitar corrupción de estado.
Control de datos tardíos con watermark
  • Usar withWatermark cuando se realizan agregaciones con ventanas.
  • Ajustar el watermark según el comportamiento real de tardanza de eventos (retrasos del sistema).
Elección del modo de salida
  • append: recomendado para ventanas con watermark y resultados finales.
  • update: útil cuando se requiere ver cambios incrementales (sinks compatibles).
  • complete: más costoso; reescribe todo el resultado en cada micro-lote, útil para demos (p. ej. consola).
Ajustes de rendimiento
  • Definir un esquema explícito para JSON (evita inferencia en streaming).
  • Ajustar spark.sql.shuffle.partitions según el tamaño del clúster y la carga.
  • Monitorizar latencia y progreso mediante StreamingQuery y logs del Driver.
  • Controlar la carga por trigger con opciones como maxOffsetsPerTrigger cuando la fuente sea Kafka.
  • Evitar demasiados ficheros pequeños ajustando particiones y frecuencia de escritura.
Gestión de offsets y reprocesamiento
  • startingOffsets=latest para consumo en tiempo real desde el momento actual.
  • startingOffsets=earliest para reprocesar histórico (si la retención de Kafka lo permite).
  • Reprocesamientos controlados se apoyan en checkpoints y en la retención del log.

Pruebas recomendadas

Un pipeline streaming debe probarse con datos normales, datos tardíos, duplicados y registros corruptos. Estas pruebas ayudan a confirmar que las ventanas, watermarks y sinks se comportan como se espera.

  • Prueba de volumen: aumentar la tasa de eventos y observar latencia.
  • Prueba de reinicio: detener la consulta y arrancarla con el mismo checkpoint.
  • Prueba de duplicados: reenviar eventos con el mismo identificador.
  • Prueba de tardanza: insertar eventos con timestamps antiguos.
  • Prueba de sink lento: simular escrituras más lentas y observar acumulación de lag.

Métricas a documentar

  • Registros procesados por segundo.
  • Duración media de micro-batch.
  • Latencia aproximada entre tiempo de evento y escritura final.
  • Número de registros válidos, inválidos y duplicados.
  • Tamaño del estado en agregaciones con ventana.
  • Número de ficheros generados en el sink.

Checklist antes de producción

  • Esquema de eventos documentado y versionado.
  • Checkpoint persistente y exclusivo por consulta.
  • Retención Kafka suficiente para recuperación.
  • Sinks idempotentes o estrategia de deduplicación.
  • Alertas sobre lag, errores de parseo y fallos de escritura.
  • Procedimiento claro para reprocesar datos.

CONCLUSIÓN

Los casos prácticos muestran cómo pasar de una integración básica Kafka-Spark a pipelines completos con ventanas, alertas, capas Bronze-Silver, checkpoints y escritura analítica.

La clave para que estos pipelines sean útiles en un entorno real está en validar datos, controlar eventos tardíos, documentar métricas y probar la recuperación ante fallos antes de depender de ellos en producción.
