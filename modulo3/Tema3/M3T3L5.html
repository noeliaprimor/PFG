ENUNCIADO

En esta práctica vas a construir una evolución avanzada del pipeline de streaming del tema anterior, en lugar de leer ficheros generados por un productor local, Spark Structured Streaming consumirá eventos reales desde Kafka. El dominio elegido es un flujo de pedidos y transacciones de comercio electrónico, con eventos válidos, eventos tardíos y registros erróneos para forzar validación y cuarentena.

El objetivo no es solo arrancar Kafka y Spark, sino diseñar un pipeline reproducible con lectura de Kafka, parseo JSON con esquema explícito, control de offsets mediante checkpoints, enriquecimiento de eventos, múltiples sinks Parquet, alertas y escritura de eventos enriquecidos en un segundo topic Kafka. El cuaderno Jupyter con el ejercicio completo y las soluciones se encuentra aquí.

Objetivos técnicos

  1. Levantar Kafka en modo KRaft y un contenedor de laboratorio con PySpark.
  2. Crear los topics orders y orders_enriched.
  3. Generar eventos JSON con un productor Python.
  4. Consumir Kafka desde Spark Structured Streaming con spark-sql-kafka-0-10.
  5. Validar eventos y separar datos correctos de datos en cuarentena.
  6. Calcular latencia de extremo a extremo y métricas por comercio, país y canal.
  7. Detectar alertas por importes altos o eventos muy tardíos.
  8. Persistir resultados en Parquet y publicar eventos enriquecidos de nuevo en Kafka.

ARQUITECTURA

El flujo del laboratorio es el siguiente: 

producer_orders.py
        |
        v
Kafka topic: orders
        |
        v
Spark Structured Streaming
        |-- output/silver_orders
        |-- output/quarantine_orders
        |-- output/merchant_metrics
        |-- output/alerts
        |
        v
Kafka topic: orders_enriched

Esquema de evento

{
  "event_id": "uuid",
  "event_time": "2026-05-11T10:30:00Z",
  "source": "orders-simulator",
  "customer_id": "c-0001",
  "merchant_id": "m-100",
  "country": "ES",
  "channel": "web",
  "amount": 125.50,
  "currency": "EUR",
  "device_id": "d-0007"
}

Sinks generados

  • silver_orders: eventos válidos enriquecidos con latencia, bucket de importe y batch.
  • quarantine_orders: eventos mal formados o incompletos con motivo de error.
  • merchant_metrics: métricas agregadas por comercio, país y canal.
  • alerts: operaciones con importe alto o latencia excesiva.
  • orders_enriched: topic Kafka con eventos válidos enriquecidos.

EJECUCIÓN GUIADA

1. Entrar en el laboratorio

cd modulo3/Tema3/kafka_spark_advanced_lab

2. Construir la imagen

docker compose build

Si hay un error de paquetes durante la construcción, reconstruye sin caché: 

docker compose build --no-cache lab

3. Arrancar Kafka

docker compose up -d kafka
docker compose ps

4. Crear topics

Crea primero el topic de entrada orders. El productor publicará aquí los eventos JSON de pedidos. 

docker compose exec kafka /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server kafka:9092 \
  --create \
  --if-not-exists \
  --topic orders \
  --partitions 3 \
  --replication-factor 1

Crea después el topic de salida orders_enriched. El job Spark escribirá aquí los eventos válidos ya enriquecidos. 

docker compose exec kafka /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server kafka:9092 \
  --create \
  --if-not-exists \
  --topic orders_enriched \
  --partitions 3 \
  --replication-factor 1

Comprueba la configuración: 

docker compose exec kafka /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server kafka:9092 \
  --describe \
  --topic orders

5. Producir eventos

docker compose run --rm lab \
  python /workspace/app/producer_orders.py \
  --bootstrap-server kafka:9092 \
  --topic orders \
  --messages 1000 \
  --sleep 0.01 \
  --invalid-rate 0.03 \
  --late-rate 0.08

6. Inspeccionar mensajes en Kafka

docker compose exec kafka /opt/kafka/bin/kafka-console-consumer.sh \
  --bootstrap-server kafka:9092 \
  --topic orders \
  --from-beginning \
  --max-messages 5

7. Ejecutar el pipeline Spark

docker compose run --rm lab \
  spark-submit \
  --packages org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.1 \
  /workspace/app/streaming_pipeline.py \
  --bootstrap-server kafka:9092 \
  --input-topic orders \
  --enriched-topic orders_enriched \
  --output /workspace/output \
  --checkpoint /workspace/checkpoint/orders_stream

La salida debe mostrar los registros válidos, inválidos, métricas y alertas procesadas por batch.

8. Comprobar el topic enriquecido

docker compose exec kafka /opt/kafka/bin/kafka-console-consumer.sh \
  --bootstrap-server kafka:9092 \
  --topic orders_enriched \
  --from-beginning \
  --max-messages 5

VALIDACIÓN

Lee los Parquet generados con Spark en modo batch: 

docker compose run --rm lab \
  python /workspace/app/validate_outputs.py

Deben aparecer registros en silver_orders, merchant_metrics y, si el productor generó importes altos o eventos tardíos, también en alerts. La cuarentena puede contener registros si se usó --invalid-rate mayor que cero.

Repetir una prueba desde cero

rm -rf output/* checkpoint/*
docker compose down -v

Después arranca de nuevo Kafka, crea los topics y repite la producción y el consumo.

AMPLIACIONES AVANZADAS

  1. Añade una agregación por ventanas de 5 minutos con watermark de 10 minutos.
  2. Calcula usuarios únicos aproximados por país y canal con approx_count_distinct.
  3. Implementa una regla de fraude por cliente: más de 5 operaciones en 2 minutos o importe acumulado superior a 1500 EUR.
  4. Cambia el productor para generar claves Kafka por customer_id y analiza la distribución por particiones.
  5. Activa compresión en el productor y compara el tiempo de envío.
  6. Ejecuta el job dos veces con el mismo checkpoint y explica por qué no reprocesa offsets.
  7. Borra solo el checkpoint, conserva Kafka, vuelve a ejecutar y documenta el reprocesamiento.
  8. Publica alertas en un tercer topic orders_alerts.
  9. Sustituye Parquet por Delta Lake y razona qué mejora en actualizaciones e idempotencia.
  10. Documenta qué decisiones son de laboratorio y cuáles serían necesarias en producción: seguridad, retención, particionado, observabilidad y gestión de errores.
